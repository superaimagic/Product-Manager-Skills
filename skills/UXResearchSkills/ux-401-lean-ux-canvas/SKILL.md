## 目的

引导产品经理创建**Jeff Gothelf的精益UX画布(v2)**——一个单页引导工具，将工作围绕**要解决的业务问题**而非**要实施的解决方案**来框定。使用它来使跨职能团队对齐核心假设、制定可测试的假设，并通过暴露理解中的差距（问题、用户、价值以及为什么解决方案应该有效）来确保每个冲刺都有学习发生。

这不是路线图或功能清单——它是一个**"保险单"**，在投入全面开发之前将假设转化为实验。画布将对话从**产出**转移到**成果**，确保团队构建正确的东西，而不仅仅是正确地构建东西。

## 核心概念

### 什么是精益UX画布？

**精益UX画布(v2)** 是一个结构化的单页模板，旨在帮助团队围绕业务问题而非解决方案框定工作。它使跨职能团队在以下方面对齐：
- 存在什么问题（以及为什么现在重要）
- 什么可衡量的成果表示成功
- 我们为谁解决
- 我们正在做出什么假设
- 我们需要先学习什么
- 哪些实验将测试这些假设

**起源：** 由Jeff Gothelf创建，《精益UX》（O'Reilly, 2013）的作者。v2版本发布是为了提高业务成果与用户成果区分的清晰度。

**核心洞察：** 画布像一个**保险单**——它在构建之前暴露理解中的差距，确保你不会把冲刺浪费在错误的事情上。

---

### 画布结构（8个框）

**布局（3列 × 3行）：**

```
┌─────────────────────┬──────────────┬───────────────────────┐
│ 1. 业务问题         │              │ 2. 业务成果           │
│                     │              │                       │
├─────────────────────┤ 5. 解决方案  ├───────────────────────┤
│ 3. 用户             │  (高框，     │ 4. 用户成果           │
│                     │   跨越第     │    和收益             │
├─────────────────────┤   1-2行)     ├───────────────────────┤
│ 6. 假设             │──────────────┤ 8. 最少工作量/        │
│                     │ 7. 先学什么  │    实验               │
│                     │              │                       │
└─────────────────────┴──────────────┴───────────────────────┘
```

**8个框（按此顺序填写）：**

1. **业务问题** — 世界中发生了什么变化，创造了一个值得解决的问题？
2. **业务成果** — 什么可衡量的行为变化表示成功？
3. **用户** — 你应该首先关注哪个用户画像？
4. **用户成果和收益** — 为什么用户会寻求这个？他们获得什么收益？
5. **解决方案** — 什么功能/倡议可能解决问题并满足用户需求？
6. **假设** — 将2-5框结合起来的可测试假设（如果/那么格式）
7. **最重要的是先学什么？** — 当前风险最高的单一假设
8. **学习下一个最重要的东西最少需要做什么？** — 验证/推翻该假设的最小实验

---

### 为什么这个工具有效

**问题优先，而非解决方案优先：**
从"世界中发生了什么变化？"开始，而不是"我们应该构建X。"这防止了解决方案驱动的思维。

**假设驱动：**
在构建之前使假设明确。每个学科暴露其风险（技术可行性、用户价值、商业可行性）。

**实验聚焦：**
在投入资源之前测试假设。小实验胜过豪赌。

**跨职能对齐：**
共享画布创建共同语言。每个人看到相同的理解差距。

---

### 关键区分（避免混淆）

**框2（业务成果）vs. 框4（用户成果）：**
- **框2：** 可衡量的**行为变化**（留存率、网站停留时间、平均订单价值）
- **框4：** **目标、收益、情感、同理心**（省钱、获得晋升、陪伴家人）

框2是指标。框4是人性。

**解决方案（框5）是假设，不是承诺：**
列出候选解决方案（功能、政策，甚至商业模式转变）。你不是承诺构建所有这些——你是在探索解决方案空间。

**假设（框6）是可测试的：**
使用模板："我们相信如果[用户]通过[解决方案]获得[收益]，[业务成果]将得以实现。"每个假设聚焦于**一个**解决方案。

---

### 反模式（这不是什么）

- **不是功能清单：** 解决方案是要测试的想法，不是待办事项列表
- **不是项目计划：** 画布框定学习，不是交付时间表
- **不是战略的替代品：** 画布执行战略；它不创造战略
- **不是一次性练习：** 随着学习而重新审视；更新假设

---

### 何时使用

✅ **使用时：**
- 开始一个新的产品计划或功能
- 重新框定一个现有项目（怀疑你在构建错误的东西）
- 在假设和实验上对齐跨职能团队
- 规划调研冲刺或MVP
- 干系人受解决方案驱动（"我们需要构建X"），你需要暴露假设

❌ **不要用时：**
- 问题和解决方案已经验证（进入执行阶段）
- 战术性bug修复或技术债务（不需要学习）
- 无论证据如何，干系人已承诺解决方案（先解决对齐问题）

它定义了：

- 进度标签（例如，背景 Qx/8 和 评分 Qx/5）
- 中断处理和暂停/恢复行为

此文件定义特定领域的评估内容。如有冲突，遵循此文件的领域逻辑。

使用 `template.md` 获取完整的填写结构。

### 第0步：收集背景

**Agent会问：**

在我们填写精益UX画布之前，先收集背景。请分享：

**业务背景：**
- 干系人需求、产品简报或计划描述
- 业务指标（收入、流失率、增长目标、KPI）
- 战略目标（OKR、路线图优先级）

**用户背景：**
- 客户研究、用户画像、待办任务洞察
- 用户反馈、支持工单、流失原因
- 竞品分析、市场趋势

**你可以粘贴：**
- PRD或产品简报
- 干系人备忘录
- 用户调研总结
- 简略描述计划

### 问题1：业务问题（框1）

**Agent会问：**

**业务有什么问题你需要解决？**

描述：
- **当前状态：** 业务今天如何交付价值？
- **什么变了：** 市场变化、竞争威胁、客户行为变化、新的交付渠道
- **为什么重要：** 为什么当前状况没有达到预期？

**好的示例：**
- "移动端流量超过桌面端后，我们的结账转化率下降了15%。我们的结账流程不是为移动端设计的，而竞品提供了一键结账。"
- "企业客户6个月后就流失了，因为我们的引导流程需要3周以上的人工配置。竞品提供自助式引导。"

**坏的示例（太模糊）：**
- "我们需要增加收入"（没有关于什么变化了的背景）
- "用户想要更多功能"（没有说明业务问题）

**Agent提供3个选项：**

1. **我会描述业务问题** — [粘贴或写下你的描述]
2. **帮我识别业务问题** — [Agent问后续问题：什么变了？什么指标受到影响？竞品在做什么？]
3. **我不确定业务问题是什么** — [Agent建议先使用 `skills/problem-statement/SKILL.md` 或 `skills/problem-framing-canvas/SKILL.md`]

**用户回应：** [选择或描述]

**Agent验证：** 这描述了什么**变了**以及为什么它产生了一个问题？如果没有，问澄清问题。

### 问题2：业务成果（框2）

**Agent会问：**

**你怎么知道你解决了业务问题？你将衡量什么？**

聚焦于**可衡量的行为变化**（欢迎使用先行指标）。问自己："如果解决方案有效，人们会做出什么不同？"

**业务成果示例：**
- 将移动端结账转化率从45%提升到60%
- 将企业引导时间从3周减少到3天
- 将平均订单价值从¥350提升到¥525
- 将客户支持工单减少30%
- 将免费转付费转化率从5%提升到10%

**重要：** 这是**框2（行为变化）**，不是框4（用户收益/同理心）。指标在这里。情感放在框4。

**Agent提供3个选项：**

1. **我有具体的指标** — [说明你的业务成果]
2. **帮我定义可衡量的成果** — [Agent根据业务问题建议指标]
3. **我只有滞后指标（收入、利润）** — [Agent帮助识别能预测这些结果的先行指标]

**用户回应：** [选择或描述]

**Agent验证：** 这些是可衡量的吗？可观察的吗？它们是否表示行为变化（不仅仅是"增加收入"）？

### 问题3：用户（框3）

**Agent会问：**

**你应该首先关注什么类型（即用户画像）的用户和客户？**

考虑：
- 谁**购买**它？
- 谁**使用**它？
- 谁**配置**它？
- 谁**管理**它？

**为什么重要：** 团队倾向于在这里走捷径（"所有人"）。画布想要的是对用户的**共享愿景**——而且不总是"客户"。

**示例：**
- "专业服务领域的中小企业主（1-10名员工）（顾问、会计师、律师）"
- "为500+名员工配置SSO的企业IT管理员"
- "每周点外卖3次以上的移动端优先的千禧一代（25-35岁）"

**Agent提供3个选项：**

1. **我已经有用户画像** — [引用 `skills/proto-persona/SKILL.md` 或粘贴用户画像]
2. **帮我识别目标用户** — [Agent问：谁经历业务问题最严重？谁最可能采用？谁最容易触达？]
3. **我需要先创建用户画像** — [Agent建议使用 `skills/proto-persona/SKILL.md` 组件技能]

**用户回应：** [选择或描述]

**Agent验证：** 这足够具体能想象一个真实的人吗？还是太宽泛（"所有用户"）？

### 问题4：用户成果和收益（框4）

**Agent会问：**

**为什么你的用户会寻求你的产品或服务？他们会获得什么收益？我们能观察到什么行为变化表明他们已经达成了目标？**

聚焦于**目标、收益、情感、同理心**——不是指标（那些放在框2）。

**用户成果和收益示例：**
- 每周节省10小时的手动数据录入时间（更多时间陪伴家人）
- 通过更快交付项目获得晋升
- 避免在朋友面前结账失败的尴尬
- 感觉有自信在不需要致电支持的情况下配置企业软件

**为什么重要：** 这是**同理心框**。关于人类动机，而不仅仅是行为变化。

**Agent提供3个选项：**

1. **我知道用户想要达成什么** — [描述用户收益]
2. **帮我识别用户收益** — [Agent提示：他们雇佣这个产品来完成什么任务？他们在避免什么痛苦？他们在寻求什么收益？]
3. **我需要先研究这些** — [Agent建议 `skills/jobs-to-be-done/SKILL.md` 或 `skills/discovery-interview-prep/SKILL.md`]

**用户回应：** [选择或描述]

**Agent验证：** 这解释了**为什么**用户关心（不仅仅是他们会做什么）吗？

### 问题5：解决方案（框5）

**Agent会问：**

**我们可以做出什么来同时解决业务问题并满足客户需求？**

列出可能有效的**功能、倡议、政策、系统，甚至商业模式转变**。鼓励广泛的解决方案空间：大/小、创新、"奇怪"的以及非技术性解决方案。

**示例：**
- 一键移动端结账（Apple Pay、微信支付）
- 自助式引导向导（无需人工配置）
- AI驱动的推荐引擎
- 管家式引导（高接触、手动的——在自动化之前先测试）
- 改变定价模式（按使用量计费代替固定费率）

**重要：** 这些是**假设**，不是承诺。你在探索选项，不是承诺构建一切。

**Agent提供3个选项：**

1. **我有解决方案想法** — [列出你的候选解决方案]
2. **帮我头脑风暴解决方案** — [Agent根据问题、成果和用户建议解决方案]
3. **我脑海中只有一个解决方案** — [Agent挑战："还有什么可以解决这个？如果那个解决方案不行怎么办？"]

**用户回应：** [选择或描述]

**Agent验证：** 你至少有3个候选解决方案吗？（更多选项 = 之后更好的假设）

### 问题6：假设（框6）

**Agent会问：**

现在让我们通过组合框2-5的假设来创建可测试的假设。

**使用此模板：**

> **我们相信** [来自框2的业务成果] **将得以实现，如果** [来自框3的用户] **通过** [来自框5的解决方案] **获得** [来自框4的收益]。

**规则：**
- 每个假设聚焦于**一个**解决方案（来自框5）
- 组合来自框2、3、4和5的假设
- 必须是可测试的（你可以设计一个实验来验证/推翻它）

**示例：**

> **我们相信** 将移动端结账转化率从45%提升到60% **将得以实现，如果** 移动端优先的千禧一代（25-35岁）**通过** 一键式微信支付集成 **获得** 更快、无摩擦的结账体验。

**Agent提供：**

基于你的输入，这里是建议的假设（框5中每个解决方案一个）：

1. [生成的假设1]
2. [生成的假设2]
3. [生成的假设3]

**选项：**
- **接受这些假设** — [Agent记录它们]
- **编辑一个假设** — [修改措辞]
- **写我自己的假设** — [使用模板]

**用户回应：** [选择或描述]

**Agent验证：** 每个假设是否清晰说明了如果解决方案有效你相信会发生什么？

### 问题7：最重要的是我们需要先学什么？（框7）

**Agent会问：**

为框6中的每个假设，识别其**风险最高的假设**。然后确定**当前风险最高的一个**。

**风险类型：**
- **价值风险：** 用户会真的使用它吗？他们在乎吗？
- **可用性风险：** 用户能搞明白怎么用吗？
- **可行性风险：** 我们在技术上能构建这个吗？
- **商业可行性风险：** 这会实现业务成果吗？

**提示：** 早期阶段，将风险聚焦于**价值**多于可行性（大多数时候）。不要构建用户不想要的东西，即使它在技术上可行。

**Agent提供：**

基于你的假设，这里是风险最高的假设：

1. **[假设1的风险]** — 例如，"用户会信任一键结账而不看逐项收费明细"
2. **[假设2的风险]** — 例如，"自助式引导将把配置时间减少到3天以内"
3. **[假设3的风险]** — 例如，"AI推荐将把平均订单价值提升50%"

**当前哪个风险最高？**

**选项：**
1. **风险1** — [选择并解释原因]
2. **风险2** — [选择并解释原因]
3. **风险3** — [选择并解释原因]
4. **我不确定哪个风险最高** — [Agent帮助排序：哪个假设如果错了，会杀死这个计划？]

**用户回应：** [选择]

**Agent记录：** 这是我们将首先测试的假设。

### 问题8：学习下一个最重要的东西最少需要做什么？（框8）

**Agent会问：**

设计一个实验来验证或推翻风险最高的假设（来自框7）**尽可能快**。

**实验类型示例：**
- **客户访谈** — 5-10个访谈来测试价值假设
- **着陆页** — 假门测试来衡量兴趣
- **管家式/手动原型** — 在自动化之前先做高接触手动版本
- **绿野仙踪** — 假装功能存在（人类在幕后）
- **烟雾测试** — 宣布功能，衡量注册量

**提醒：** 实验设计是一门艺术。画布推动你去做——预期在实验设计上单独迭代。

**Agent提供：**

基于你风险最高的假设，这里是实验选项：

1. **客户访谈** — 与10个用户交谈，问他们是否会使用[解决方案]，观察他们当前的变通方案
2. **着陆页测试** — 创建一个描述[解决方案]的着陆页，衡量注册率
3. **管家式测试** — 为5个用户手动交付[收益]，观察他们是否达到[成果]
4. **绿野仙踪原型** — 用幕后人类假装[解决方案]，看用户是否完成工作流
5. **设计自定义实验** — [描述你的实验]

**你将运行哪个实验？**

**用户回应：** [选择一个或描述自定义实验]

## 常见陷阱

### 1. **从解决方案而非问题开始**
**失败模式：** 框1说"我们需要构建X"而不是描述什么变了。

**后果：** 你构建的是某人已经决定好的解决方案，而没有验证问题是否存在。

**修复：** 问："世界中有什么变化？为什么这现在是一个问题（相对于6个月前）？"

---

### 2. **模糊的业务成果**
**失败模式：** 框2说"增加收入"或"让用户开心"。

**后果：** 没有办法衡量成功；无法判断实验是否有效。

**修复：** 定义可衡量的行为变化。"将平均订单价值从¥350提升到¥525"或"将支持工单减少30%。"

---

### 3. **过于宽泛的用户群体**
**失败模式：** 框3说"所有用户"或"每个人"。

**后果：** 无法设计有目标的实验；在不采用画像上浪费时间。

**修复：** 选择一个用户画像开始。你以后可以扩展。

---

### 4. **混淆框2和框4**
**失败模式：** 把情感放在框2，把指标放在框4（或反之）。

**后果：** 假设不对齐；成功标准不清晰。

**修复：** **框2 = 行为变化（指标）。框4 = 目标、收益、情感（同理心）。**

---

### 5. **框5只有一个解决方案**
**失败模式：** 因为干系人已经决定而只列出一个功能。

**后果：** 没有探索替代方案；无法测试哪个解决方案最好。

**修复：** 强迫自己列出3+个解决方案。问："还有什么可以解决这个问题？"

---

### 6. **跳过实验（框8）**
**失败模式：** "我们就直接构建，看看会发生什么。"

**后果：** 花费数周/数月构建错误的东西。

**修复：** 先设计最小实验。如果你想不出来，使用 `skills/pol-probe-advisor/SKILL.md` 来选择验证方法。

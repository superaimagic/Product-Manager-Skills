## 目的

定义和记录一个**生命证明(PoL)探针**——一个轻量级、可丢弃的验证产物，旨在在昂贵开发之前暴露残酷真相。当你需要消除特定风险或测试窄假设**而不构建生产级软件**时使用此技能。PoL探针是侦察任务，不是MVP——它们注定被删除，而非被扩展。

此框架防止原型剧场（昂贵演示让利益相关者印象深刻但什么也没教），并迫使你将验证方法匹配到实际学习目标。

## 核心概念

### 什么是PoL探针？

一个**生命证明(PoL)探针**是一个有目的的、可丢弃的验证实验，旨在尽可能便宜和快速地回答一个具体问题。它不是产品，不是MVP，不是试点——它是一个有针对性的真相探寻任务。

**起源：** 由Dean Peters（Productside）提出，基于Marty Cagan 2014年的原型风味工作和Jeff Patton的原则：*"测试你的想法最昂贵的方式是构建生产级软件。"*

---

### 5个核心特征

每个PoL探针必须满足以下标准：

| 特征 | 含义 | 为什么重要 |
|------|------|------------|
| **轻量级** | 最小资源投入（几小时/几天，而非几周） | 如果昂贵，当数据显示应该终止时你会回避 |
| **可丢弃** | 明确计划删除，而非扩展 | 防止沉没成本谬误和范围蔓延 |
| **窄范围** | 测试一个具体假设或风险 | 宽泛实验产生模糊结果 |
| **残酷诚实** | 暴露残酷真相，而非虚荣指标 | 礼貌的数据是无用的数据 |
| **微小且聚焦** | 侦察任务，绝非MVP | 小表面面积 = 更快的学习循环 |

**反模式：** 如果你的"原型"感觉太精美而不舍得删除，那它不是PoL探针——而是原型剧场。

---

### PoL探针 vs. MVP

| 维度 | PoL探针 | MVP |
|------|---------|-----|
| **目的** | 通过窄假设测试来降低决策风险 | 证明想法或捍卫路线图方向 |
| **范围** | 单一问题、单一风险 | 最小可交付产品增量 |
| **生命周期** | 几小时到几天，然后删除 | 几周到几个月，然后迭代 |
| **受众** | 内部团队 + 窄用户样本 | 生产环境中的真实客户 |
| **保真度** | 足够逼真以捕获信号 | 生产级（或接近） |
| **结果** | 了解什么*不*奏效 | 了解什么*奏效*（并交付它） |

**关键区别：** PoL探针是**MVP前的侦察**。你跑探针来决定*是否*应该构建MVP，而非发布什么东西。

---

### 5种原型风味

将探针类型匹配到你的假设，而非你的工具舒适度。

| 类型 | 核心问题 | 时间线 | 工具/方法 | 何时使用 |
|------|----------|--------|----------|----------|
| **1. 可行性检查** | "我们能构建这个吗？" | 1-2天 | GenAI提示链、API测试、数据完整性扫描、探测-删除代码 | 技术风险未知；第三方依赖不明确 |
| **2. 任务聚焦测试** | "用户能否无摩擦地完成这项工作？" | 2-5天 | Optimal Workshop、UsabilityHub、任务流 | 关键时刻（字段标签、决策点、流失区域）需要验证 |
| **3. 叙事原型** | "这个工作流能否赢得利益相关者的认可？" | 1-3天 | Loom演练、Sora/Synthesia视频、幻灯片故事板 | 你需要"讲述而非测试"——分享故事，衡量兴趣 |
| **4. 合成数据模拟** | "我们能否在不触及生产环境的情况下建模？" | 2-4天 | Synthea（用户模拟）、DataStax LangFlow（提示逻辑测试） | 边缘案例探索；未知-未知项暴露 |
| **5. 氛围编码PoL探针** | "这个方案能否经受真实用户接触？" | 2-3天 | ChatGPT Canvas + Replit + Airtable = "缝合软件" | 你需要用户对工作流/UX的反馈，但不需要生产级代码 |

**黄金法则：** *"用最便宜的原型揭示最残酷的真相。如果不刺痛，那可能只是剧场。"*

---

### 何时使用PoL探针

✅ **在以下情况使用PoL探针：**
- 你有具体的、可证伪的假设要测试
- 某个特定风险阻碍你的下一个决策（技术可行性、用户任务完成、利益相关者支持）
- 你需要快速得到残酷真相（几天内，而非几周）
- 构建生产软件是不成熟的或浪费的
- 你能在开始之前描述"失败"的样子

❌ **在以下情况不要使用PoL探针：**
- 你试图给高管留下印象（那是原型剧场）
- 你已经知道答案只是想要验证（那是确认偏见）
- 你无法阐述明确的假设或处置计划
- 学习目标太宽泛（"客户会喜欢这个吗？"）
- 你用它来回避艰难决策

## 应用

使用 `template.md` 获取完整的填入结构。

### PoL探针模板

使用此结构来记录你的探针：

```markdown
# PoL探针：[描述性名称]

## 假设
[一句话陈述你相信为真的内容]
示例："如果我们将引导表单减少到3个字段，完成率将超过80%。"

## 消除的风险
[你要解决的特定风险或未知项是什么？]
示例："我们不知道用户是否会因为表单太长而放弃注册。"

## 原型类型
[选择5种风味中的一种]
- [ ] 可行性检查
- [ ] 任务聚焦测试
- [ ] 叙事原型
- [ ] 合成数据模拟
- [x] 氛围编码PoL探针

## 目标用户 / 受众
[谁会与此探针交互？]
示例："10位来自我们早期访问候补名单的用户，非技术型中小企业老板。"

## 成功标准（残酷真相）
[你在寻求什么真相？什么能证明你错了？]
- **通过：** 8+用户在2分钟内完成注册
- **失败：** <6用户完成，或平均时间超过5分钟
- **学到：** 识别具体的流失字段

## 工具 / 技术栈
[你将用什么来构建这个？]
示例："ChatGPT Canvas用于表单UI，Airtable用于数据捕获，Loom用于会后访谈。"

## 时间线
- **构建：** 2天
- **测试：** 1天（10个用户会话）
- **分析：** 1天
- **处置：** 第5天（删除所有代码，保留学习文档）

## 处置计划
[你何时以及如何删除这个？]
示例："用户会话完成后，归档录制，删除缝合软件代码，在飞书中记录学习。"

## 负责人
[谁对运行和处置此探针负责？]

## 状态
- [ ] 假设已定义
- [ ] 探针已构建
- [ ] 用户已招募
- [ ] 测试已完成
- [ ] 学习已记录
- [ ] 探针已处置
```

---

### 质量检查清单

在启动PoL探针之前，验证：

- [ ] **轻量级：** 你能在1-3天内构建这个吗？
- [ ] **可丢弃：** 你已承诺处置日期了吗？
- [ ] **窄范围：** 它测试一个假设吗？
- [ ] **残酷诚实：** 如果你错了，数据会让你痛苦吗？
- [ ] **微小且聚焦：** 这比MVP更小吗？
- [ ] **可证伪：** 你能描述"失败"的样子吗？
- [ ] **明确负责人：** 有一个人对执行和处置此探针负责吗？

如果任何答案是"否"，修改你的探针或重新考虑你是否需要它。

## 常见陷阱

- 运行宽泛的"用户会喜欢这个吗？"实验，而非测试一个可证伪的假设
- 把PoL探针当作proto-MVP且拒绝处置
- 使用回避不适真相的虚荣指标
- 在测试开始前跳过预定义失败阈值
- 先选工具再选假设

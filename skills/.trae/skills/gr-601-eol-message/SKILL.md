## 目的
撰写清晰、富有同理心的产品生命周期结束（EOL）消息，传达产品或功能的停用决定，解释背后原因，说明客户影响，提供迁移支持，并定位替代解决方案。用于在困难的产品过渡中维护客户信任，通过展现关怀和提供明确的迁移路径来减少客户流失。

这不是一份通用的下架公告——而是一份以客户为中心的沟通，在承认损失的同时将变化表述为进步。

## 核心概念

### EOL消息框架
一份有效的EOL消息需要在坦诚面对变化的同时，对客户影响抱有同理心。它包含以下要素：

1. **公司背景：** 你是谁以及你对客户的承诺
2. **公告内容：** 什么将被停用，什么将替代它
3. **原因解释：** 为什么这个决定对客户有益（而不仅仅是对业务有益）
4. **当前产品背景：** 该产品曾是什么，曾服务于谁
5. **客户影响：** 这将如何影响用户（承认带来的不便）
6. **迁移方案：** 替代品是什么，以及它如何在旧产品基础上做出改进
7. **支持措施：** 你将如何帮助客户完成迁移
8. **时间表：** 关键日期和里程碑
9. **行动号召：** 后续步骤和联系方式

### 为什么这一框架有效
- **同理心优先：** 在解释决定之前先承认客户面临的不便
- **清晰明确：** 关于什么将改变、何时改变，没有含糊
- **支持导向：** 表明你不会在迁移过程中抛弃客户
- **面向未来：** 将变化表述为进步，而非损失

### 反模式（这不是什么）
- **不是一份简短的下架通知：** "我们将停用产品X。再见。"
- **不是以业务为中心：** 不要以"这会降低我们的成本"开头
- **不是模糊表述：** "很快"不是一个时间表
- **不是防御性口吻：** 不要责怪客户（"低使用率迫使我们关闭"）

### 何时使用
- 停用产品、功能或服务
- 将客户从旧平台迁移到新平台
- 下架收购目标的产品
- 弃用某项技术栈或API

### 何时不使用
- 针对小的功能调整（不要对小变化过度沟通）
- 在你制定好迁移计划之前（在你知道如何支持客户之后再沟通）
- 如果你暗自希望客户不会注意到（保持透明）

## 应用

使用 `template.md` 获取完整的填写结构。

### 第一步：收集背景信息
在撰写之前，确保你已具备以下信息：
- **被停用的产品：** 具体什么将被终止？
- **替代方案：** 什么将替代它（如果有的话）？
- **时间表：** 关键日期（公告日、功能冻结日、停用日、数据导出截止日）
- **客户影响：** 有多少用户受影响？哪些工作流程会被打断？
- **支持计划：** 迁移支持、培训、折扣、数据导出工具
- **原因：** 为什么会发生？（技术过时、战略转移、整合等）

**如果缺少背景信息：** 在拥有完整的迁移计划之前，不要发送消息。客户会问"我现在该怎么办？"——你必须有一个答案。

---

### 第二步：起草产品过渡叙事

#### 公司背景
确立你是谁以及你的承诺：

```markdown
### 产品过渡叙事

**我们是谁：** [描述公司及其与即将淘汰的产品的关系]
- [关于公司对客户承诺的关键点]
- [关于公司产品演进的关键点]
- [关于公司未来愿景的关键点]
```

**示例：**
```markdown
**我们是谁：** 慧工科技，一家服务于50,000家中小企业的智能工作流自动化平台
- 我们致力于帮助你节省时间，专注于重要事项
- 我们根据你的反馈和技术进步持续演进产品
- 我们正朝着一个自动化触手可及、强大且简单的未来前进
```

---

#### 公告
清晰直接：

```markdown
**公告：**
- [用一句话清晰声明产品的EOL，并介绍其替代品]
```

**示例：**
- "我们将在2026年12月31日停用慧工经典版，并将所有客户迁移至慧工专业版。"

---

#### 原因解释（以客户利益为中心）
解释为什么这对客户有益：

```markdown
**因为：**
- [原因1：例如，技术进步]
- [原因2：例如，性能提升]
- [原因3：例如，更好地匹配客户需求]

**对你而言这意味着：**
- [从客户角度描述影响和收益]
```

**示例：**
```markdown
**因为：**
- 慧工经典版运行在陈旧的基础设施上，限制了性能和可扩展性
- 慧工专业版基于现代技术构建，可实现更快的自动化、更好的集成和实时协作
- 整合至一个平台使我们能将100%的工程资源投入到你所需要的功能上

**对你而言这意味着：**
- 自动化执行速度提升3倍
- 50多种与你已在使用的工具的新集成
- 获得实时协作和移动应用等新功能
```

---

### 第三步：提供当前产品背景

承认失去的东西：

```markdown
### 当前产品背景

**我们的产品** [被停用产品的名称]
- **是一款** [简要描述产品及其主要功能]
- **已经服务于** [目标客户/用户] 长达 [时间长度]
- **通过提供** [产品提供的关键收益或解决方案]
```

**示例：**
```markdown
**我们的产品** 慧工经典版
- **是一款** 帮助企业消除重复性任务的智能工作流自动化工具
- **已经服务于** 超过20,000家客户，历时8年
- **通过提供** 可靠、简单易用的自动化方案，无需技术背景即可上手
```

---

### 第四步：承认客户影响

坦诚面对带来的不便：

```markdown
### 客户影响

**我们理解这可能会在以下方面影响您：**
- [对客户运营或流程的潜在影响1]
- [对客户运营或流程的潜在影响2]
- [对客户运营或流程的潜在影响3（如适用）]
```

**示例：**
```markdown
**我们理解这可能会在以下方面影响您：**
- 需要花费时间将工作流从经典版迁移到专业版
- 学习新功能和界面变化
- 如果您对工作流进行了定制化，需要更新集成或API连接
```

---

### 第五步：呈现过渡方案

使用定位声明格式（参考 `skills/positioning-statement/SKILL.md`）：

```markdown
### 过渡方案

**面向** [受EOL影响的目标客户/用户]
- **目前使用** [被淘汰产品的名称]
- [替代产品名称]
- **是一款** [替代产品品类的定义]
- **能够** [关于用户收益的陈述，聚焦于连续性和改进]

### 差异化与延续性

- **如同** [被淘汰产品]，
- [替代产品名称]
- **提供** [替代品如何保持旧产品的关键收益]
- **同时还提供** [新的收益或改进]
```

**示例：**
```markdown
### 过渡方案

**面向** 中小企业主
- **目前使用** 慧工经典版
- 慧工专业版
- **是一款** 下一代智能工作流自动化平台
- **能够** 保持你所喜爱的所有简单性和可靠性，同时提供3倍更快的性能、50多种新集成和实时协作功能

### 差异化与延续性

- **如同** 慧工经典版，
- 慧工专业版
- **提供** 无需编程即可构建自动化流程、可靠的执行和透明的定价
- **同时还提供** 3倍更快的工作流、移动应用访问、实时团队协作，以及与飞书、钉钉、企业微信等工具的集成
```

---

### 第六步：概述支持措施和时间表

#### 支持措施

```markdown
### 支持措施与后续步骤

**为确保顺利过渡，我们将：**
- [支持措施1，例如，"为所有客户提供一对一的迁移协助"]
- [支持措施2，例如，"自动迁移您的工作流（需经您批准）"]
- [支持措施3，例如，"为现有客户提供慧工专业版3个月的折扣"]
```

---

#### 时间表

```markdown
### 时间表

- [关键日期1及相关里程碑，例如，"2026年3月1日：迁移工具上线"]
- [关键日期2及相关里程碑，例如，"2026年9月1日：慧工经典版变为只读"]
- [关键日期3及相关里程碑，例如，"2026年12月31日：慧工经典版完全停用，数据导出截止日期"]
```

**质量检查：**
- **充足的准备时间：** 客户需要时间来规划（通常为6-12个月）
- **明确的里程碑：** 功能何时冻结？何时完全关闭？
- **数据导出截止日期：** 他们何时会失去对其数据的访问权限？

---

### 第七步：提供明确的后续步骤

```markdown
### 行动号召

- [客户明确的后续步骤，例如，"登录您的账户以启动迁移向导"]
- [问题或协助的联系方式，例如，"联系我们的支持团队：support@huigong.com 或拨打400-888-XXXX"]
```

## 常见陷阱

### 陷阱一：以业务为中心的原因
**症状：** "我们正在停用产品X以降低成本和整合产品组合。"

**后果：** 客户感觉自己是商业决策中的附带受害者。

**修复：** 将原因围绕客户利益来表述："我们正在整合到产品Y，以便将100%的资源投入到你所需求的功能上。"

---

### 陷阱二：模糊的时间表
**症状：** "产品X将很快停用。"

**后果：** 客户无法做规划。焦虑和流失增加。

**修复：** 提供具体日期："3月1日：迁移工具上线。12月31日：完全关闭。"

---

### 陷阱三：没有支持计划
**症状：** "您需要迁移到产品Y。祝您好运！"

**后果：** 客户感觉被抛弃。高流失风险。

**修复：** 提供迁移支持："一对一协助、自动迁移工具、3个月折扣、培训资源。"

---

### 陷阱四：忽视客户影响
**症状：** 消息从公告直接跳到"这是新产品！"

**后果：** 客户感觉自己的关切没有被回应。

**修复：** 明确承认影响："我们理解这需要时间来进行迁移和学习新功能。"

---

### 陷阱五：简短或防御性的语气
**症状：** "由于低使用率，我们将关闭产品X。"

**后果：** 听起来像是在责怪客户。

**修复：** 保持同理心和前瞻性："我们正在整合资源，聚焦于自动化的未来。"

## 目的

通过自适应提问引导产品经理创建用户故事地图——关于系统、用户、工作流和优先级——然后生成二维地图，包含主干（活动）、用户任务和发布切片。用于从扁平待办列表转向视觉故事地图，传达大局、识别缺失功能、实现有意义的发布规划——避免故事脱离整体系统叙事的"无上下文碎片"问题。

这不是待办列表生成器——这是一个视觉沟通框架，按用户工作流（水平）和优先级（垂直）组织工作。

## 核心概念

### 什么是用户故事地图？

故事地图（Jeff Patton）在**二维**中组织用户故事：

**横轴（左到右）：** 活动按叙事/工作流顺序排列——你向某人解释系统时使用的顺序

**纵轴（上到下）：** 每个活动内的优先级，最核心的任务在顶部

**结构：**
```
主干（横向顶部的活动）
↓
用户任务（按优先级纵向下降）
↓
细节/验收标准（底部）
```

### 核心原则

**主干：** 核心活动构成系统的结构核心——它们之间不做优先排序；它们是叙事流。

**行走骨架：** 所有活动中最高优先级的任务构成最小可行产品——最小端到端功能。

**肋骨：** 支撑任务在每个活动下纵向下降，通过位置表示优先级。

**从左到右、从上到下构建策略：** 在所有主要功能上增量构建，而非在开始另一个之前完全完成一个。

### 为什么这样有效
- **视觉沟通：** 故事地图保持作为信息辐射器展示，维持对大局的关注
- **叙事结构：** 按用户工作流组织，不是技术架构
- **发布规划：** 水平切片揭示MVP和增量发布
- **缺口识别：** 揭示扁平待办列表遮蔽的缺失功能

### 反模式（这不是什么）
- **不是甘特图：** 故事地图展示优先级，不是时间估算
- **不是技术架构：** 地图遵循用户工作流，不是系统层（UI → API → DB）
- **不是项目计划：** 它是发现和沟通工具，不是进度表

### 适用场景
- 启动新产品或重大功能
- 重构现有待办列表（从扁平列表转向视觉地图）
- 在范围和优先级上对齐利益相关者
- 规划MVP或增量发布

### 不适用场景
- 单功能项目（故事地图过度）
- 待办列表已清晰理解和优先排序时
- 技术重构工作（没有用户工作流可映射）

## 应用方法

本交互技能提出**最多5个自适应问题**，每步提供**3-4个带编号选项**。

使用`template.md`获取引导议程和输出检查清单。

交互模式：当需要一步一问流程并在决策点有编号推荐、常规问题有快速选择选项时，配合`skills/workshop-facilitation/SKILL.md`使用。如果用户要求一次性输出，跳过多轮引导。

### 步骤0：收集上下文（提问之前）

**Agent建议：**

在创建故事地图之前，让我们收集上下文：

**产品/功能上下文：**
- 你在映射什么系统或功能？
- 产品概念、PRD草稿或现有待办列表
- 网站文案、定位材料或用户流程
- 现有用户故事（如果从扁平待办列表过渡）

**用户上下文：**
- 目标人物画像或用户细分
- 用户研究、访谈或旅程地图
- 待完成任务或问题陈述

**你可以直接粘贴这些内容，或简要描述系统。**

---

### 问题1：定义范围

**Agent问：**
"你在映射什么？（范围是什么？）"

**提供4个带编号的选项：**

1. **整个产品** — "从发现到完成的完整端到端系统"（常见于新产品或全面重写）
2. **重大功能领域** — "更大产品中的特定工作流（例如'注册引导'、'支付结算'、'报表')"（常见于功能上线）
3. **用户旅程** — "特定用户目标或待完成任务（例如'招聘外包人员'、'在线报税')"（常见于JTBD驱动的映射）
4. **重构/重设计** — "正在重建或简化现有产品/功能"（常见于遗留系统现代化）

**或描述你的特定范围。**

**用户回答：** [选择或自定义]

---

### 问题2：识别用户/人物画像

**Agent问：**
"这张地图的主要用户是谁？（列出人物画像或用户细分。）"

**提供4个带编号的选项：**

1. **单一人物画像** — "一个主要用户类型（例如'小企业主')"（简化映射，适合MVP）
2. **多人物画像，共享工作流** — "不同用户类型，相同核心活动（例如'买家'和'卖家'都浏览列表)"（常见于市场平台）
3. **多人物画像，不同工作流** — "不同用户类型有不同工作流（例如'管理员' vs. '终端用户')"（需要分开的地图或泳道）
4. **组织内的角色** — "不同职能（例如'产品经理'、'设计师'、'工程师')"（常见于内部工具）

**或描述你的用户。**

**自适应：** 使用步骤0中提供的上下文的人物画像（原初画像、JTBD等）

**用户回答：** [选择或自定义]

---

### 问题3：生成主干（活动）

**Agent说：**
"让我们构建主干——用户为达成目标所执行活动的叙事流。"

**Agent根据范围（Q1）和用户（Q2）生成5-8个活动**，从左到右按工作流顺序排列。

**示例（如果范围 = "电商结算流程"）：**

```
主干活动（从左到右）：

1. 浏览商品
2. 加入购物车
3. 查看购物车
4. 填写配送信息
5. 填写支付信息
6. 确认订单
7. 收到确认通知
```

**Agent问：**
"这个主干是否捕捉了完整工作流？我们需要添加、删除或重新排列活动吗？"

**用户回答：** [确认、修改或添加自定义活动]

---

### 问题4：生成用户任务（每个活动下）

**Agent说：**
"现在让我们在每个活动下添加用户任务，按优先级排列（顶部 = 必须有，底部 = 锦上添花）。"

**Agent为每个活动生成3-5个用户任务**，纵向按优先级排列。

**示例（活动2："加入购物车"）：**

```
加入购物车（活动）
├─ 添加单品到购物车（必须有，行走骨架）
├─ 调整数量（必须有）
├─ 一次添加多件（应该有）
├─ 保存商品稍后购买（锦上添花）
└─ 添加礼品包装（锦上添花）
```

**Agent为所有主干活动重复此过程**，展示完整地图。

**Agent问：**
"这是否捕捉了关键任务？优先级是否正确（顶部 = MVP，底部 = 后续发布）？"

**用户回答：** [确认、修改或添加自定义任务]

---

### 问题5：识别发布切片（行走骨架 + 增量）

**Agent说：**
"让我们通过在地图上画水平线来定义发布切片。"

**Agent生成3个发布切片：**

**发布1（行走骨架）：** 所有活动中最高优先级的任务——最小端到端功能

**发布2（下一增量）：** 第二优先级的任务，增强核心工作流

**发布3（打磨/扩展）：** 第三优先级的任务（锦上添花、边缘场景、优化）

**示例：**

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
发布1（行走骨架）：
- 浏览商品（基础列表视图）
- 添加单品到购物车
- 查看购物车（明细 + 总额）
- 填写配送信息（姓名、地址）
- 填写支付信息（仅银行卡）
- 确认订单（基础确认）
- 收到邮件确认
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
发布2（增强）：
- 商品筛选/搜索
- 调整购物车数量
- 保存稍后购买
- 多种配送选项
- 多种支付方式
- 订单追踪链接
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
发布3（打磨）：
- 商品推荐
- 游客结算
- 礼品包装
- 优惠券
- 高级支付选项
- 购后调研
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

**Agent问：**
"这些发布切片是否合理？我们需要调整范围或优先级吗？"

**用户回答：** [确认或修改]

## 示例

### 示例1：好的故事地图（电商结算流程）

**Q1回答：** "重大功能领域 — 电商结算工作流"

**Q2回答：** "单一人物画像 — 在线购物者"

**Q3 - 主干生成：**
```
浏览 → 加入购物车 → 查看购物车 → 填写配送 → 填写支付 → 确认 → 收到确认
```

**Q4 - 用户任务生成：**

```
浏览商品
├─ 查看商品列表 (R1)
├─ 搜索/筛选 (R2)
└─ 商品推荐 (R3)

加入购物车
├─ 添加单品 (R1)
├─ 调整数量 (R2)
└─ 保存稍后购买 (R3)

查看购物车
├─ 查看明细 + 总额 (R1)
├─ 应用优惠券 (R2)
└─ 估算配送费 (R3)

[...等等...]
```

**Q5 - 发布切片：**
- **发布1：** 行走骨架——基础流程无附加功能
- **发布2：** 搜索、数量调整、优惠券
- **发布3：** 推荐、游客结算、礼品选项

**为什么这有效：**
- 主干遵循用户叙事（不是技术层）
- 行走骨架交付端到端价值
- 增量发布增加 sophistication 而不破坏核心流程

---

### 示例2：差的故事地图（技术层）

**主干（错误）：**
```
UI层 → API层 → 数据库层 → 部署
```

**为什么这失败：**
- 不是以用户为中心（用户不在乎技术架构）
- 无法增量交付端到端价值
- 瀑布思维伪装成故事映射

**修正：**
- 按用户工作流映射："注册 → 配置设置 → 邀请团队 → 开始项目"
- 每个发布交付完整工作流，不是单个层

## 常见误区

### 误区1：伪装的扁平待办列表
**症状：** 故事地图只是一个纵向列表，没有水平叙事

**后果：** 丧失沟通价值；仍然是"无上下文碎片"

**修正：** 强制水平结构——活动横跨顶部，任务纵向下降

---

### 误区2：技术架构作为主干
**症状：** 主干 = "前端 → 后端 → 数据库"

**后果：** 不是以用户为中心，无法增量交付价值

**修正：** 主干应遵循用户工作流，不是系统层

---

### 误区3：功能完整的瀑布
**症状：** 发布1 = "完整构建活动1"，发布2 = "完整构建活动2"

**后果：** 直到所有活动完成才有端到端价值

**修正：** 行走骨架 = 跨所有活动的薄切片，增量增强

---

### 误区4：过早过多细节
**症状：** 尝试预先映射每个边缘场景和验收标准

**后果：** 分析瘫痪，丢失大局

**修正：** 从主干+高层任务开始，后续细化

---

### 误区5：地图藏在工具中
**症状：** 故事地图存在于Jira/Miro中，从不展示

**后果：** 丧失作为信息辐射器的价值

**修正：** 物理打印/张贴地图；让团队每天可见

## 目标（Purpose）

Sprint 计划是敏捷开发中最重要的会议之一，但在中国互联网团队里也是最容易走形的：

- 产品列了 20 个需求，希望「这个 Sprint 全做完」
- 研发说「来不及」，但又不知道怎么定量说明
- Sprint 目标变成了任务列表：「这两周把这 8 个需求做完」
- Sprint 结束：60% 完成率，大家都不满意

这个技能提供一套**轻量、实用、适合中国互联网团队的 Sprint 规划框架**，核心是：先定目标，再选故事，再评估容量。

## 核心概念（Key Concepts）

### Sprint 基本参数

**Sprint 长度选择：**

| Sprint 长度 | 适用场景 | 注意事项 |
|------------|---------|---------|
| 1 周 | 早期阶段、快速验证 | 会议开销比例高，适合小团队 |
| 2 周（推荐） | 大多数互联网产品团队 | 平衡灵活性和稳定性 |
| 3-4 周 | 技术复杂度高、B2B 产品 | 周期过长，反馈循环慢 |

**团队速度（Velocity）：**
- 定义：团队在一个 Sprint 内平均能完成的故事点数
- 如何建立：跑 2-3 个 Sprint 后统计实际完成的故事点
- 使用原则：以历史平均速度规划，不以理想值规划

**故事点（Story Points）：**
- 度量的是「相对复杂度」，不是「绝对工时」
- 斐波那契序列：1, 2, 3, 5, 8, 13, 21（超过 13 的故事需要拆分）
- 团队对齐：「3 分」对所有人的含义要一致

### Sprint 三层结构

```
Sprint 目标（Sprint Goal）
  — 这个 Sprint 结束后，我们要实现什么用户价值？
  — 一句话，清晰，可验证
  — 示例：「让新用户在首次注册后 10 分钟内完成第一次核心操作」

    └─ Sprint Backlog（Sprint 待办列表）
         — 为实现 Sprint 目标而选择的用户故事
         — 按优先级排序
         — 总故事点数 ≤ 团队速度

           └─ 每个故事的任务拆解
                — 研发将故事拆分为具体技术任务
                — 每个任务 ≤ 8 小时工时
                — 认领到人
```

### 产能计算（关键！）

中国互联网团队经常出现超负荷承诺，根本原因是**没有计算实际可用产能**：

```
理论产能 = 团队人数 × Sprint 天数 × 每天工作小时

实际可用产能 = 理论产能
  - 会议时间（Sprint 计划 + 站会 + Sprint 评审 + 复盘 = 约 0.5 天/周）
  - Bug 修复（通常占 15-25%）
  - 紧急需求缓冲（互联网节奏下建议留 10-15%）
  - 个人假期/调休

实际产能系数 ≈ 0.6-0.7
```

**示例：**
```
团队：4 名后端 + 2 名前端 + 1 名测试（7 人）
Sprint 长度：10 个工作日
理论产能：7 × 10 × 8 = 560 小时
实际可用：560 × 0.65 = 364 小时 ≈ 折算为故事点参考值
```

### 反模式（Anti-Patterns）

❌ **「Sprint 目标 = 任务列表」**
- 错误：「本次 Sprint 完成：用户中心改版、消息推送、数据报表模块」
- 正确：「本次 Sprint：让用户能看到完整的个人数据看板（含基础消息通知）」

❌ **「产品排满，研发硬撑」**
- 错误：排了 50 点的故事，团队实际速度是 30 点
- 后果：开发加班、测试时间压缩、质量下降、团队士气受损
- 解法：以历史速度为上限，不以「希望」为上限

❌ **「需求在 Sprint 中途突然加入」**
- 错误：Sprint 第 5 天，产品说「有个紧急需求要加进来」
- 后果：Sprint 目标被破坏，已承诺的故事无法完成
- 解法：除非 P0 线上故障，其他需求等下个 Sprint；紧急需求有 15% 缓冲池

❌ **「没有 Definition of Done（完成定义）」**
- 错误：开发说「做完了」，但没有联调、没有测试、没有产品验收
- 解法：团队共同定义 DoD，每个故事都要过 DoD 才算完成

## 应用方法（Application）

### Sprint 计划会流程（2-3 小时）

**参与者：** 产品经理 + 技术负责人 + 开发团队 + 测试

---

**阶段一：确认 Sprint 目标（30 分钟）**

产品经理提出：
> 「下个 Sprint，我们最重要的事是什么？用一句话说。」

讨论原则：
- Sprint 目标应该是「业务价值/用户价值」，不是「功能清单」
- 一个 Sprint 只有一个主目标，可以有一个次要目标
- 技术团队要认同目标，不只是「接受任务」

Sprint 目标模板：
```
让 [谁] 能够 [做什么]，从而 [达到什么结果]。
```

示例：
- ✅「让核心用户能在 App 内完成全部报告导出操作，无需切换到 PC 端」
- ❌「完成报告导出功能和 PC 端兼容性优化」

---

**阶段二：梳理 Sprint Backlog（60 分钟）**

**步骤 1：从产品 Backlog 选择用户故事**

产品经理提前准备：
- 下个 Sprint 候选故事列表（按优先级排序）
- 每个故事已有验收标准（AC）
- 估算过的故事点（或在会上当场估算）

**步骤 2：故事点估算（Planning Poker 或直接讨论）**

Planning Poker 流程（每个故事 3-5 分钟）：
1. 产品经理读故事内容 + AC
2. 每人选择一张故事点卡（1/2/3/5/8/13/21）
3. 同时亮牌
4. 分歧最大的两人说明原因
5. 重新估算，达成共识

注意：
- 故事点 > 13：这个故事太大了，需要拆分
- 有人喊「需要澄清才能估算」：先澄清，再估算

**步骤 3：确认容量上限**

```markdown
本次 Sprint 容量计算：

成员：[姓名 1]（全程）/ [姓名 2]（周三请假 1 天）/ ...
Sprint 天数：10 个工作日
会议开销：- 8 小时（计划会 2h + 评审 1h + 复盘 1h + 每日站会 10min×10=）
Bug 修复缓冲：- 15%
合理容量目标：[团队历史速度] 故事点

本次选入故事总点数：[X 点]（不超过合理容量目标）
```

---

**阶段三：故事拆解为任务（30-60 分钟，研发主导）**

每个进入 Sprint 的故事，研发团队将其拆解为具体技术任务：

```markdown
故事：用户可以在 App 内导出 PDF 报告

技术任务拆解：
- [ ] 后端：新增 PDF 生成接口（估时：8h，负责人：小明）
- [ ] 后端：PDF 模板设计和数据填充（估时：6h，负责人：小明）
- [ ] 前端：导出按钮 UI 实现（估时：4h，负责人：小王）
- [ ] 前端：导出进度提示和错误处理（估时：3h，负责人：小王）
- [ ] 测试：编写测试用例（估时：3h，负责人：小李）
- [ ] 测试：联调测试（估时：4h，负责人：小李）

总计：28h（在团队容量内）
```

---

**阶段四：风险识别与确认（15 分钟）**

问题清单：
1. 这个 Sprint 有技术未知量吗？（如新技术、未做过的集成）
2. 有跨团队依赖吗？（等其他团队的 API、设计稿、数据）
3. 有人在 Sprint 中途有假期或重要外部会议？

对每个风险：
- 是否会影响 Sprint 目标？
- 应对预案是什么？

---

### Sprint 过程中的规则

**每日站会（15 分钟，站着开）**

每人回答三个问题：
1. 昨天完成了什么？
2. 今天计划做什么？
3. 有什么障碍？（障碍立刻记录，会后处理）

注意：站会不是汇报会，不讨论解决方案，只暴露障碍

**Sprint 中途需求变更处理**

```
紧急程度评估：
├─ P0（线上故障/重大问题）→ 立刻处理，从 Sprint 中移除等量故事
├─ P1（紧急但不影响现有功能）→ 加入紧急缓冲池，Sprint 内评估是否排入
└─ P2/P3 → 记录到 Product Backlog，下个 Sprint 排期
```

**Sprint 燃尽图（Burndown Chart）**

每天更新剩余故事点，如果第 5 天后曲线仍在理想线之上，需要预警。

---

### Sprint 评审会（Sprint Review）

**时长：** 1 小时
**参与者：** 团队 + 利益相关方（产品老板、运营、重要干系人）

议程：
1. 演示已完成的功能（只展示「完成定义」通过的故事）
2. 收集反馈（不是辩论，只是记录）
3. 更新 Product Backlog（根据反馈调整后续优先级）

**不要演示：**
- 未完成的功能（会制造期望）
- Bug 频发的功能（应该回到 Sprint 继续修）

---

### Definition of Done（完成定义）模板

团队定义，一次定好，每个故事都适用：

```markdown
# 完成定义（DoD）v1.0

代码层面：
- [ ] 代码已通过同行评审（PR Review）
- [ ] 单元测试覆盖率 > 60%（新增代码）
- [ ] 无已知 P0/P1 Bug

质量层面：
- [ ] 测试完成冒烟测试（Happy Path 通过）
- [ ] 测试完成边界场景测试
- [ ] 产品经理完成验收

部署层面：
- [ ] 已部署到测试环境
- [ ] 已在测试环境验证通过

文档层面：
- [ ] API 文档更新（如有接口变更）
- [ ] 涉及用户可见变化的，已通知运营/客服
```

## 示例（Examples）

### 示例：某电商 App 的双周 Sprint 规划

**背景：** 6 人开发团队（3 后端 + 2 前端 + 1 测试），双周 Sprint，历史速度 28 点

**Sprint 目标（讨论后确认）：**
> 「让用户能在手机端完成购物车 → 结算 → 支付的完整流程，不再需要切换到 PC」

**Sprint Backlog：**

| 用户故事 | 故事点 | 优先级 |
|---------|-------|------|
| 用户可以查看购物车商品和价格 | 3 | P0 |
| 用户可以修改购物车数量/删除商品 | 5 | P0 |
| 用户可以选择收货地址 | 3 | P0 |
| 用户可以选择支付方式（微信/支付宝） | 5 | P0 |
| 用户可以完成支付并收到确认通知 | 8 | P0 |
| 用户可以查看订单状态 | 3 | P1 |
| **总计** | **27 点** | |

**容量评估：**
- 历史速度 28 点，本 Sprint 有 1 人请假 1.5 天
- 调整后目标：27 点（在合理范围内）

**风险识别：**
- 支付宝 API 集成是新功能，技术风险较高 → 预留额外缓冲，实在不行降级为「先只支持微信支付」

## 常见陷阱（Common Pitfalls）

### 陷阱 1：Sprint 目标太模糊

**症状：** Sprint 目标是「提升用户体验」「优化性能」「做好基础架构」

**后果：** 团队不知道优先级，Sprint 结束也无法判断是否成功

**解法：** Sprint 目标必须具体到「用户能做什么」或「业务指标提升多少」

---

### 陷阱 2：「产品排需求，研发没有说不的权利」

**症状：** 产品单方面排满 Sprint，研发只能被动接受

**后果：** 研发加班→质量差→更多 Bug→下个 Sprint 更乱

**解法：** Sprint 容量由研发团队确认，不由产品经理决定；产品决定「做什么」，研发决定「能做多少」

---

### 陷阱 3：不做 Sprint 复盘（Retrospective）

**症状：** 每个 Sprint 有同样的问题，但从来不系统解决

**后果：** 团队效率无法提升，同样的错误重复犯

**解法：** 每个 Sprint 结束后 1 小时复盘：哪些做得好、哪些需要改、下个 Sprint 改变 1 件事

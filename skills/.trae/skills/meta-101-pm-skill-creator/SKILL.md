## 目的

以交互方式遍历完整的技能设计过程——从原始想法或内容到结构化的、仓库合规的SKILL.md草案。提出自适应问题来确定技能类型、范围、结构和内容，然后生成一个可供验证的草案。

本技能是 `skill-authoring-workflow` 的对话补充。那个技能定义了过程和验证关卡。本技能与你坐在一起，在你构建之前弄清楚要构建什么。

## 核心概念

### 何时使用本技能 vs. 其他创作工具

| 工具 | 最适用场景 |
|------|-----------|
| **本技能（skill-creator）** | 你有一个想法或原始内容，需要通过对话帮助塑造成正确的结构 |
| `skill-authoring-workflow` | 你已经知道要构建什么，需要过程检查清单和验证步骤 |
| `scripts/build-a-skill.sh` | 你知道结构，想要一个终端向导来收集各部分 |
| `scripts/add-a-skill.sh` | 你有一个源文档，想要端到端的AI辅助生成 |

### 三种技能类型（决策标准）

- **组件：** 一个产出物或模板。自成一体。被其他技能引用。问："这是一个人们创建的东西吗？"
- **交互式：** 带有自适应问题和枚举建议的引导式对话。问："这需要来回对话才有效用吗？"
- **工作流：** 引用其他技能的多阶段编排。问："这跨多个步骤协调多项活动吗？"

### 技能结构（不可省略的部分）

每个技能必须按顺序包含以下部分：
1. **目的** — 它做什么 + 何时使用它（以成果为导向）
2. **核心概念** — 框架、定义、心智模型
3. **应用** — Agent可以遵循的分步指示
4. **示例** — 至少一个具体的、详细的示例
5. **常见陷阱** — 命名的失败模式，附带后果和修复方法
6. **参考资料** — 相关技能、外部框架、源材料

### 元数据约束

- `name`：小写kebab-case，≤ 64个字符
- `description`：≤ 200个字符，必须包含触发线索（"Use when..."）
- `intent`：较长的仓库级摘要（无字符限制）
- `type`：`component`、`interactive`、`workflow`之一

### 引导方式参考来源

使用 [`workshop-facilitation`](../workshop-facilitation/SKILL.md) 作为本技能的默认交互协议。

它定义了：
- 会话预告 + 进入模式（引导式、内容转储、最佳猜测）
- 每次一问的轮流对话，使用通俗易懂的提示
- 进度标签（例如，上下文 Q1/5）
- 中断处理和暂停/恢复行为
- 决策点上的编号建议
- 常规问题的快选编号响应选项（在有用时包含 `其他（请说明）`）

本文件定义了领域特定的内容。如有冲突，以本文件的领域逻辑为准。

## 应用

这个交互式技能提出**最多5个自适应问题**，然后交付一个**完整的SKILL.md草案**，包含frontmatter、所有必需部分和仓库合规结构。

---

### 步骤0：会话开始

**Agent说：**

"我将帮助你从头设计一个新的PM技能。这大约需要10-15分钟，最多5个问题。你想如何开始？

1. **引导式** — 我将逐题提问，基于你的回答构建技能（推荐）
2. **内容转储** — 粘贴你的原始内容、笔记或框架，我将提出一个技能结构
3. **最佳猜测** — 只告诉我主题，我将起草一个你可以细化的技能"

---

### 问题1：原始素材是什么？（Q1/5）

**Agent询问：**

"我们将什么转化为一个技能？给我你拥有的任何东西。"

1. **一个框架或心智模型** — 一种思考PM问题的结构化方式（例如，优先级矩阵、决策树）
2. **一个模板或产出物** — PM创建的交付物（例如，PRD部分、定位声明、Epic格式）
3. **一个流程或工作流** — 完成一项PM任务的多步骤方法（例如，探索Sprint、路线图规划）
4. **一个辅导或咨询主题** — PM需要引导式、自适应帮助的领域（例如，利益相关者导航、定价决策）

**或用你自己的话描述你拥有的东西。**

**Agent注：** 如果用户粘贴原始内容而非选择选项，分析内容并推断答案。在继续之前确认你的解读。

---

### 问题2：技能类型决策（Q2/5）

**基于Q1的回答，Agent推荐一个类型并确认：**

*如果Q1 = 框架或模板：*

"这听起来像一个**组件技能**——一个自成体系的产出物或参考。它将包含一个模板、质量标准和示例。这符合吗，还是我漏掉了一个对话式/自适应的元素？

1. **是的，组件** — 它是一个独立的交付物或参考
2. **实际上，它需要对话** — 用户需要引导式问题来很好地使用它（→ 交互式）
3. **它比一个产出物更大** — 它编排多个步骤或其他技能（→ 工作流）"

*如果Q1 = 流程或工作流：*

"这听起来像一个**工作流技能**——一个多阶段的流程。它是引用或编排其他离散技能/产出物，还是更像一个引导式对话？

1. **是的，工作流** — 它有不同的阶段，可能引用其他技能
2. **它更像对话式的** — 价值来自自适应问题和建议（→ 交互式）
3. **它比我描述的更简单** — 实际上它是一个带步骤的产出物（→ 组件）"

*如果Q1 = 辅导或咨询：*

"这听起来像一个**交互式技能**——引导式问题，根据回答自适应调整，以枚举建议结束。听起来对吗？

1. **是的，交互式** — 它需要来回对话才有效用
2. **它更像一个参考** — 用户只需要框架，不需要对话（→ 组件）
3. **它是一个完整的流程** — 多阶段，编排其他技能（→ 工作流）"

---

### 问题3：范围和命名（Q3/5）

**Agent询问：**

"我们应该如何命名这个技能？我需要两样东西：

**a) 一个工作名称** — 小写kebab-case，最多64个字符（例如，`feature-investment-advisor`、`user-story`、`discovery-process`）

**b) 一句话描述** — 它做什么 + 何时使用它。必须适配在200个字符内，并以动词或'Use when...'开头

给我你最好的尝试，如果需要我会收紧它。或者只是描述技能的目的，我将两个都提出来。"

**Agent注：** 在继续之前验证名称格式（kebab-case，≤ 64字符）和描述长度（≤ 200字符）。如果任一未通过，建议修复。

---

### 问题4：关键内容（Q4/5）

**此问题根据Q2的技能类型自适应调整：**

*对于组件技能：*

"这个产出物或框架的核心元素是什么？我需要：
1. **模板或结构** — 它包含哪些部分/字段？
2. **质量标准** — 什么区分好坏？
3. **一个具体的示例** — 一个展示它做得好的填写版本

给我你拥有的任何东西——要点、粗略笔记或完整草稿。"

*对于交互式技能：*

"带我走过对话流程：
1. **开始问题是什么？** — 用户首先需要告诉你什么？
2. **2-4条分支路径是什么？** — 回答如何改变接下来发生的事情？
3. **什么建议会出现？** — 你会提供的3-5个结果是什么？

告诉我你看到的决策树——即使它是粗略的。"

*对于工作流技能：*

"映射出各个阶段：
1. **主要步骤是什么？**（通常3-6个阶段）
2. **每个阶段的输入和输出是什么？**
3. **这引用了哪些现有技能？**（用 `scripts/find-a-skill.sh --keyword <主题>` 检查）
4. **决策点在哪里？**

给我流程——顺序的、分支的，或两者都有。"

---

### 问题5：陷阱和边缘情况（Q5/5）

**Agent询问：**

"当人们做得不好时，什么会出错？我需要2-3个失败模式：

1. **命名失败** — 给它一个标签（例如，'指标表演'、'英雄综合征'）
2. **描述后果** — 当某人落入这个陷阱时会发生什么？
3. **陈述修复方法** — 纠正措施是什么？

如果你不确定，告诉我你见过的最常见错误，我将帮助构建它。"

---

### 草案生成

在收集完Q1-Q5的回答后，Agent生成一个完整的SKILL.md草案，包括：

1. **YAML frontmatter** — `name`、`description`、`intent`、`type`、`best_for`、`scenarios`、`estimated_time`
2. **目的** — 从Q1 + Q3合成
3. **核心概念** — 从Q4内容结构化
4. **应用** — 从Q4派生的分步指示
5. **示例** — 来自Q4的具体示例（或如果未提供则生成）
6. **常见陷阱** — 从Q5结构化
7. **参考资料** — 对话中识别的相关技能 + 源材料

**Agent在生成后说：**

"这是你的SKILL.md草案。在我们说完成之前：

1. **评审草稿** — 它捕捉了你的意图吗？
2. **运行验证** — `python3 scripts/check-skill-metadata.py skills/<名称>/SKILL.md`
3. **检查触发** — `python3 scripts/check-skill-triggers.py skills/<名称>/SKILL.md --show-cases`
4. **冒烟测试** — `./scripts/test-a-skill.sh --skill <名称> --smoke`

想要我调整什么，还是你准备好验证了？"

## 示例

### 示例：框架 → 组件技能

**Q1：** "1 — 一个框架。它是一个用于评估构建vs购买决策的2x2矩阵。"
**Q2：** "1 — 是的，组件。"
**Q3：** "名称：`build-vs-buy-matrix`。描述：'使用2x2矩阵评估构建vs购买决策，维度为战略价值和实现复杂度。适用于在内部构建和购买解决方案之间选择时。'"
**Q4：** "轴是战略差异化（低/高）和实现复杂度（低/高）。四个象限：购买（低/低）、合作（低/高）、构建（高/低）、投资（高/高）。质量标准：每个象限有清晰的决策规则和一个示例。"
**Q5：** "最大错误：将其视为一次性决策。后果：你构建了某样东西，市场变了，你被困在维护非差异化技术上。修复：每季度重新评估。"

**输出：** 完整的组件技能，包含模板、填写示例和季度评审陷阱。

---

### 示例：咨询主题 → 交互式技能

**Q1：** "4 — 辅导主题。帮助PM驾驭他们第一次与领导层的定价对话。"
**Q2：** "1 — 是的，交互式。"
**Q3：** "名称：`pricing-conversation-advisor`。描述：'为第一次与领导层的定价对话做准备。适用于向高管呈现定价变化或新变现方案时。'"
**Q4：** "开始问题：定价变化是什么？分支：新产品定价、现有产品涨价、免费转付费转换、按用量过渡。建议：要带的数据、异议处理、框架策略。"
**Q5：** "失败模式：'以数字开场'。后果：高管只需关注价格点而非战略。修复：在揭示数字之前始终以业务案例和客户影响开场。"

**输出：** 完整的交互式技能，包含3个问题的流程、4个分支和每个分支的枚举建议。

---

### 反模式：范围蔓延

**Q1：** "我想要一个涵盖所有产品战略的技能——愿景、路线图、OKR、优先级排序、利益相关者对齐和指标。"

**Agent响应：** "那是4-6个独立的技能，不是一个。让我们选择现在最重要的一个，构建那个。哪一块最紧急？

1. 愿景 → 组件技能
2. 路线图规划 → 工作流技能（已存在：`roadmap-planning`）
3. 优先级排序 → 交互式技能（已存在：`prioritization-advisor`）
4. 其他

从一项开始。我们可以之后再构建其他的。"

## 常见陷阱

### 陷阱1：类型混淆
**症状：** 将某个东西称为"工作流"，但实际它是一个带编号步骤的模板。

**后果：** 过度设计的技能，引用不存在的子技能并混淆用户。

**修复：** 诚实地应用Q2中的类型决策问题。如果它不编排其他技能，也没有不同的阶段和不同的输入/输出，它可能是一个组件。

### 陷阱2：不触发的描述
**症状：** 描述说明了技能是什么（"一个定价决策框架"），但没有说何时使用它。

**后果：** Claude无法将用户请求匹配到该技能。它闲置不用。

**修复：** 每个描述必须回答"Use when..."——例如，"使用结构化框架评估定价决策。适用于在定价模型之间选择或准备定价提案时。"

### 陷阱3：跳过验证
**症状：** "草案看起来不错，我们发布吧。"

**后果：** 破损的frontmatter、缺失的部分、失败的交叉引用、不一致的目录计数。

**修复：** 在认为技能完成之前，始终运行 `check-skill-metadata.py` 和 `check-skill-triggers.py`。没有例外。

### 陷阱4：什么都想装的范畴
**症状：** 试图将整个领域塞进一个技能。

**后果：** 一个500行的怪物，什么都做不好。太宽泛无法准确触发，太长无法有用。

**修复：** 一个技能 = 一项任务。如果你需要超过6个应用步骤或超过4个分支，你可能需要多个技能。

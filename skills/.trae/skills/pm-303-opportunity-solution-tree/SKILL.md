## 目的

通过从利益相关者请求中提取目标结果，生成机会选项（要解决的问题），映射潜在解决方案，并根据可行性、影响和适销性选择最佳概念验证（POC），指导产品经理创建机会解决方案树（OST）。用于从模糊的产品请求转变为结构化的发现，确保团队在跳到解决方案之前解决正确的问题——避免"功能工厂"综合症和想法的过早收敛。

这不是路线图生成器——它是结构化的发现过程，输出带有可测试解决方案假设的验证机会。

## 核心概念

### 什么是机会解决方案树（OST）？

OST是一个视觉框架（Teresa Torres，*Continuous Discovery Habits*），连接：
1. **期望结果**（商业目标或产品指标）
2. **机会**（客户问题、需求、痛点或愿望，可以推动结果）
3. **解决方案**（解决每个机会的方法）
4. **实验**（验证解决方案的测试）

**结构：**
```
         期望结果（1）
                 |
     +-----------+-----------+
     |           |           |
机会      机会        机会（3）
     |           |           |
   +-+-+       +-+-+       +-+-+
   | | |       | | |       | | |
 S1 S2 S3    S1 S2 S3    S1 S2 S3（总共9个解决方案）
```

### 为什么这个方法有效

- **结果驱动：** 从商业目标开始，而不是功能请求
- **在收敛之前发散：** 在挑选解决方案之前探索多个机会
- **以问题为中心：** 机会是客户面临的问题，而不是伪装成问题的解决方案
- **可测试：** 每个解决方案映射到实验，而不仅仅是"构建并发布"
- **POC选择：** 在承诺资源之前评估可行性、影响、适销性

### 反模式（这不是什么）

- **不是功能列表：** 机会是客户面临的问题，而不是"我们需要深色模式"
- **不是解决方案优先：** 不要从"我们应该构建X"开始——从"客户在Y上挣扎"开始
- **不是瀑布规划：** OST是发现工具，不是项目计划
- **不是一次性练习：** OST随着你从实验中学习而演进

### 何时使用

- 利益相关者请求功能或产品计划
- 为新产品的领域启动发现
- 澄清模糊的OKR或战略目标
- 优先级排序首先要解决什么问题
- 在你要驱动什么结果上对齐团队

### 何时不使用

- 当问题已经验证时（转到解决方案测试）
- 对于战术错误修复或技术债务（不需要发现）
- 当利益相关者要求特定解决方案时（先解决对齐问题）

## 应用

使用 `template.md` 获取完整的填写结构。

此外互式技能遵循**两阶段过程**：

**阶段1：** 生成OST（提取结果、识别机会、映射解决方案）
**阶段2：** 选择POC（评估解决方案、推荐最佳起点）

---

### 步骤0：收集上下文（在问题之前）

**代理建议：**

在我们创建你的机会解决方案树之前，让我们收集上下文：

**利益相关者请求或产品计划：**
- 利益相关者要求什么？（功能请求、产品想法、战略目标）
- 任何现有材料：PRD草案、OKR文档、策略备忘录、会议笔记
- 问题陈述、客户投诉或研究发现

**产品上下文（如果可用）：**
- 网站文案、定位陈述、产品描述
- 竞争对手材料、客户评论（G2、Capterra）、社区讨论
- 使用数据、支持工单、流失原因

**你可以直接粘贴此内容，或简要描述请求。**

## 阶段1：生成机会解决方案树

### 问题1：提取期望结果

**代理询问：**
"此计划的期望结果是什么？（你试图移动什么商业或产品指标？）"

**提供4个枚举选项：**

1. **收入增长** — "增加ARR、扩展现有客户的收入、新收入流"（对于扩展产品常见）
2. **客户留存** — "减少流失、增加激活、改善参与/粘性"（对于具有留存问题的成熟产品常见）
3. **客户获取** — "增加注册、试用转换、新用户增长"（对于早期阶段或增长产品常见）
4. **产品效率** — "减少支持成本、减少价值时间、改善运营指标"（对于优化运营的成熟产品常见）

**或描述你的特定期望结果（可衡量：例如，"将试用到付费的转换从15%提高到25%"）。**

**用户响应：** [选择或自定义]

**代理提取并确认：**
- **期望结果：** [特定、可衡量的结果]
- **为什么这很重要：** [来自利益相关者请求或上下文的理由]

---

### 问题2：识别机会（要解决的问题）

**代理基于期望结果和提供的上下文生成3个机会。**

**代理说：**
"根据你的期望结果（[来自Q1]）和你提供的上下文，这里是**3个机会**（客户问题或需求），可以推动此结果："

**示例（如果结果 = 增加试用到付费的转换）：**

1. **机会1：用户在试用期间未体验价值** — "新用户注册但未完成入门，从未达到'啊哈时刻'，在看到核心价值之前放弃"
   - 证据：[来自上下文：入门analytics、支持工单、退出调查]

2. **机会2：定价不清晰或不对齐** — "用户不确定付费计划是否值得;不理解他们为价格得到什么;定价页面令人困惑"
   - 证据：[来自上下文：转换漏斗在定价页面下降、销售异议]

3. **机会3：免费计划'足够好'** — "用户无限期地留在免费层，因为它满足他们的需求；没有令人信服的理由升级"
   - 证据：[来自上下文：freemium用户留存数据、支持工单要求变通办法]

**代理询问：**
"哪个机会感觉最关键的首先探索，或者你想修改/添加机会吗？"

**用户响应：** [选择或自定义]

---

### 问题3：为选定的机会生成解决方案

**代理为选定的机会生成3个解决方案想法。**

**代理说：**
"对于**机会[X]：[名称]**，这里是**3个潜在解决方案**来测试："

**示例（如果选择机会1：用户在试用期间未体验价值）：**

1. **解决方案1：引导式入门清单** — "交互式清单，逐步引导用户完成核心工作流（例如，'创建你的第一个项目'、'邀请队友'、'完成任务'）"
   - 假设：结构化指导增加完成率
   - 实验：A/B测试清单 vs. 无清单，测量激活率

2. **解决方案2：价值时间触发器** — "当用户卡住时自动提示（例如，'你还没有创建项目——这里有一个模板来开始'）"
   - 假设：主动提示防止流失
   - 实验：跟踪与提示的参与，测量试用到付费的提升

3. **解决方案3：人工辅助的入门** — "为高价值的试用用户提供15分钟的入门电话与CSM（企业、团队计划）"
   - 假设：个人接触增加高意图用户的转换
   - 实验：提供给50个试用用户，测量与控制组的转换

**代理询问：**
"你想为另一个机会探索解决方案，还是转到POC选择？"

**用户响应：** [选择或自定义]

## 阶段2：选择概念验证（POC）

生成解决方案后，代理帮助选择首先要测试哪个解决方案。

---

### 问题4：评估解决方案（可行性、影响、适销性）

**代理说：**
"让我们评估你的解决方案以选择最佳起点。我将根据**可行性**（构建有多难）、**影响**（它能移动结果多少）和**适销性**（与客户需求的匹配程度）给每个解决方案评分。"

**代理生成评估表：**

| 解决方案 | 可行性（1-5） | 影响（1-5） | 适销性（1-5） | 总分 | 理由 |
|----------|-------------------|--------------|------------------|-------------|-----------|
| 解决方案1：引导式入门清单 | 4 | 4 | 5 | 13 | 高可行性（UI模式）、证明的影响（行业标准）、强大的适销性（用户期望入门） |
| 解决方案2：价值时间触发器 | 3 | 3 | 4 | 10 | 中等可行性（需要analytics集成）、中等影响（取决于触发器质量）、良好的适销性 |
| 解决方案3：人工辅助的入门 | 5 | 5 | 3 | 13 | 高可行性（无开发工作）、高影响（个人接触）、较低的适销性（不扩展，仅高接触） |

**评分标准：**
- **可行性：** 1 = 数月的工作，5 = 天/周
- **影响：** 1 = 最小结果移动，5 = 主要结果转变
- **适销性：** 1 = 客户不关心，5 = 客户积极请求这个

**代理推荐：**

**推荐的POC：解决方案1（引导式入门清单）**

**为什么这个POC：**
- 平衡可行性（可在2-4周内构建）与影响（证明的模式）
- 强大的适销性（用户期望入门指导）
- 可测试的假设："如果我们引导用户完成核心工作流，激活率将从X%增加到Y%"

**替代POC：** 解决方案3（人工辅助的入门）
- 如果你想在没有开发工作的情况下快速学习，从这里开始
- 与20-50个试用用户手动测试，收集定性反馈
- 使用学习通知自动化解决方案（解决方案1）

**不推荐用于POC：** 解决方案2（价值时间触发器）
- 需要更多前期投资
- 影响取决于触发器质量（需要实验以正确获得）
- 作为验证核心入门流程后的后续更好

---

### 问题5：定义实验

**代理询问：**
"你将如何测试此解决方案？实验是什么？"

**提供3个枚举选项：**

1. **A/B测试** — "构建MVP，显示给50%的试用用户，比较转换 vs. 控制"（最适合：定量验证，需要流量）
2. **原型 + 可用性测试** — "创建可点击的原型，观看10个用户尝试入门，收集定性反馈"（最适合：早期阶段验证，低流量）
3. **手动礼宾测试** — "与20个用户手动运行解决方案（例如，亲自引导他们完成入门），测量结果"（最适合：快速学习，无开发工作）

**或描述你的实验方法。**

**用户响应：** [选择或自定义]

## 常见陷阱

### 陷阱1：伪装成解决方案的机会

**症状：** "机会：我们需要移动应用"

**后果：** 在探索问题之前，你已经收敛到解决方案。

**修复：** 将机会重新框定为客户问题："移动优先用户在旅途中无法访问产品。"

---

### 陷阱2：跳过发散（跳到单个解决方案）

**症状：** "我们知道解决方案是[X]，只需要构建它"

**后果：** 错过更好的替代方案，没有学习。

**修复：** 每个机会生成至少3个解决方案。在收敛之前强制发散。

---

### 陷阱3：结果太模糊

**症状：** "期望结果：改善用户体验"

**后果：** 无法衡量成功，无法优先级排序机会。

**修复：** 使结果可衡量："将NPS从30提高到50"或"将入门流失从60%降低到40%。"

---

### 陷阱4：无实验（只是构建它）

**症状：** 选择解决方案并直接转到路线图

**后果：** 无验证，构建错误事物的高风险。

**修复：** 每个解决方案必须映射到实验。无实验 = 无OST。

---

### 陷阱5：分析瘫痪（永远探索）

**症状：** 生成20个机会、50个解决方案，从不选择一个

**后果：** 团队卡在发现中，没有进展。

**修复：** 限制为3个机会，每个3个解决方案（总共9个）。选择POC，运行实验，学习，迭代。

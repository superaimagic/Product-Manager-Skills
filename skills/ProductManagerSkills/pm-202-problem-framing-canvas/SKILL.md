## 目的
通过结构化提问引导产品经理完成MITRE问题框架画布过程，跨越三个阶段：向内看（审视自己的假设和偏见）、向外看（理解谁经历了问题以及谁没有）、重新框定（将洞察综合为可执行的问题陈述和"我们如何可能"问题）。使用此技能确保你在跳入解决方案之前解决正确的问题——避免确认偏见、遗漏的利益相关者和方案优先思维。

这不是解决方案头脑风暴——它是一个问题框定工具，拓宽视角、挑战假设、产生清晰的、公平导向的问题陈述。

## 核心概念

### 什么是MITRE问题框架画布？

问题框架画布（MITRE创新工具包，v3）是一个结构化框架，帮助团队在提出解决方案之前全面探索问题空间。它分为**三个区域**：

1. **向内看** — 审视自己的假设、偏见以及你可能是问题的一部分
2. **向外看** — 理解谁经历了问题、谁从中受益以及谁被遗漏了
3. **重新框定** — 将洞察综合为清晰的、可执行的问题陈述和"我们如何可能"问题

### 画布结构

```
┌─────────────────────────────────────────────────────────────────┐
│ 向内看                                                           │
│ - 问题是什么？（症状）                                            │
│ - 为什么我们没有解决它？（新的、困难的、优先级低等）              │
│ - 我们如何是问题的一部分？（假设、偏见）                          │
└─────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────┐
│ 向外看                                                           │
│ - 谁经历了这个问题？何时/何地/后果？                              │
│ - 还有谁有这个问题？谁没有？                                     │
│ - 谁被遗漏了？                                                  │
│ - 问题存在/不存在时谁受益？                                     │
└─────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────┐
│ 重新框定                                                         │
│ - 换种方式说，问题是：[重新表述]                                 │
│ - 我们如何可能[行动]，以实现[目标]？                             │
└─────────────────────────────────────────────────────────────────┘
```

### 为什么这有效
- **拓宽视角：** 强迫你超越自己的假设看
- **公平导向：** 关注边缘化声音并问"谁被遗漏了？"
- **挑战偏见：** 在框定问题之前要求明确审视假设
- **可执行输出：** 产生准备用于解决方案探索的HMW陈述

### 反模式（这不是什么）
- **不是解决方案头脑风暴：** 画布框定问题；解决方案之后来
- **不是功能请求列表：** 聚焦底层问题，而非表面症状
- **不是一个人的练习：** 需要多元视角来挑战群体思维

### 何时使用此技能
- 开始新计划的发现工作
- 重新框定现有问题（怀疑你在解决错误的东西）
- 在构建解决方案之前挑战假设
- 对齐跨职能团队关于问题定义

### 何时不使用此技能
- 问题已经被充分理解和验证
- 战术性bug修复或技术债务（不需要深层框定）
- 利益相关者已经承诺了某个解决方案（先处理对齐问题）

## 应用

使用 `template.md` 获取完整的填入结构。

此交互式技能遵循**三阶段流程**，在每个阶段提出**自适应问题**。

---

### 步骤0：收集上下文（提问之前）

**Agent建议：**

在我们框定你的问题之前，让我们收集上下文：

**问题上下文：**
- 初始问题陈述或利益相关者需求
- 你观察到的症状（支持工单、流失数据、用户投诉）
- 现有研究（用户访谈、调查、分析）
- 你对问题的假设

**利益相关者上下文：**
- 谁受这个问题影响？（用户、客户、内部团队）
- 谁要求解决这个问题？（高管、销售、客户）
- 谁可能被遗漏了？

**你可以直接粘贴这些内容，或简要描述问题。**

## 第一阶段：向内看

**目标：** 审视自己的假设、偏见以及你可能是问题的一部分。

---

### 问题1：问题是什么？（描述症状）

**Agent问：**
"你目前理解的问题是什么？描述症状。"

**提供4个编号选项：**

1. **客户痛点** — "客户在[具体任务/成果]上挣扎"（例如"客户找不到他们需要的功能"）
2. **业务指标问题** — "我们看到[指标下降]"（例如"上季度流失增加了15%"）
3. **利益相关者需求** — "利益相关者说我们需要[功能/改变]"（例如"销售团队说我们需要更好的报告"）
4. **观察到的行为** — "我们注意到[模式/趋势]"（例如"用户在引导的第3步放弃"）

**或描述你的问题/症状。**

**用户响应：** [选择或自定义]

**Agent提取：**
- **问题（初始框定）：** [来自用户的描述]

---

### 问题2：为什么我们没有解决它？

**Agent问：**
"为什么这个问题还没有被解决？"

**提供6个编号选项（可多选）：**

1. **是新的** — "问题最近才出现"
2. **是困难的** — "技术上复杂或资源密集"
3. **优先级低** — "其他计划优先"
4. **缺乏资源** — "预算、人员或时间不够"
5. **缺乏权限** — "无法做出决策或获得认可"
6. **系统性不公平** — "问题不成比例地影响边缘化群体，被忽视了"

**或描述你自己的原因。**

**用户响应：** [选择或自定义]

**Agent捕获：**
- **解决的障碍：** [原因列表]

---

### 问题3：我们如何是问题的一部分？（假设与偏见）

**Agent问：**
"你（或你的团队）如何可能是问题的一部分？你带来了什么假设或偏见？"

**提供4个编号选项：**

1. **假设我们知道客户想要什么** — "我们没有与真实用户验证"（确认偏见）
2. **为自己而非用户优化** — "构建对我们容易的，而非对他们有价值的"（内部偏见）
3. **遗漏特定用户细分** — "聚焦多数用户，忽略边缘案例或边缘化群体"（幸存者偏见）
4. **方案优先思维** — "在理解根本问题之前就跳到'我们需要[功能X]'"（过早收敛）

**或描述你的具体假设/偏见。**

**用户响应：** [选择或自定义]

**Agent捕获：**
- **需要挑战的假设：** [偏见列表]

**目标：** 理解谁经历了问题、谁从中受益以及谁被遗漏了。

### 问题4：谁经历了这个问题？（何时、何地、后果）

**Agent问：**
"谁经历了这个问题？他们何时何地经历它？他们面临什么后果？"

**Agent提示用户描述：**

- **何地：** 物理或数字位置（例如"移动应用"、"企业部署"）
- **后果：** 对用户的影响（例如"每周浪费2小时"、"错过截止日期"、"流失"）

**适配：** 使用上下文中的角色（原型角色、JTBD、客户研究）

### 问题6：谁被遗漏了？谁受益？

**Agent问：**
"到目前为止谁被遗漏在了对话之外？当这个问题存在或不存在时谁受益？"

**Agent提示：**
- **谁被遗漏了：** 边缘化声音、边缘案例、被忽视的利益相关者

**示例：**
- "引导流程有问题时谁受益？" → "销售团队不需要支持复杂工作流；工程不需要构建引导流程"

### 问题7：重新表述问题

**Agent说：**
"基于我们探索的一切，让我们以新的方式重新表述问题。"

**Agent使用第一、二阶段的洞察生成一个精炼的问题陈述：**

**模板：**
"问题是：[谁]难以[完成什么]，因为[根本原因]，导致[后果]。这影响了[特定细分]，并且一直被忽视，因为[第一阶段的偏见/假设]。"

**示例（SaaS引导）：**
"问题是：非技术型小企业老板难以在引导过程中激活我们的产品，因为我们使用术语密集的UI且缺乏引导式工作流，导致60%在24小时内放弃。这不成比例地影响了没有技术支持的个体创业者，并且一直被忽视，因为我们的团队为企业用户优化——他们有IT部门。"

**Agent问：**
"这个重新表述是否捕捉了核心问题？我们应该精炼它吗？"

**用户响应：** [批准或修改]

### 问题8：创建"我们如何可能"陈述

**Agent说：**
"现在让我们用'我们如何可能'陈述使其可执行。"

**模板：**
"我们如何可能[解决问题的行动]，以实现[目标/期望条件]？"

**示例（SaaS引导）：**
"我们如何可能用通俗易懂的提示引导非技术用户完成引导过程，以将激活率从40%提升到70%？"

**Agent问：**
"这个HMW陈述是否设置了正确的解决方案空间？我们应该调整吗？"

## 常见陷阱

### 陷阱1：跳过"向内看"（假设你是中立的）
**症状：** 团队直接跳到"向外看"而不审视偏见

**后果：** 群体思维持续存在，假设未被挑战

**修复：** 强制明确讨论假设和偏见（Q2-Q3）

---

### 陷阱2：忽略"谁受益"问题
**症状：** 画布完成但没有探索谁从问题存在中受益

**后果：** 错过政治动态、变革阻力

**修复：** 总是问"如果这个问题被解决了谁会损失？"（Q6）

---

### 陷阱3：泛泛的问题陈述
**症状：** 重新框定的问题模糊（"改善用户体验"）

**后果：** HMW陈述不可执行

**修复：** 使问题具体（谁、什么、何时、后果、根本原因）

---

### 陷阱4：HMW陈述太窄
**症状：** "我们如何可能添加一个移动应用？"

**后果：** 将解决方案空间限制为一个想法

**修复：** 保持HMW宽泛："我们如何可能让移动优先用户在任何设备上访问核心工作流？"

---

### 陷阱5：独自练习（没有多元视角）
**症状：** PM独自填写画布

**后果：** 偏见持续存在，边缘化声音仍然被遗漏

**修复：** 用跨职能团队 + 客户输入引导画布工坊

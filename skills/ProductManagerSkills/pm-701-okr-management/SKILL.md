## 目标（Purpose）

引导产品经理完成 OKR 全生命周期管理：从制定有野心的目标和可衡量的关键结果，到跨层级对齐、持续追踪和季度复盘。

OKR 不是绩效考核工具——它是**方向对齐和聚焦工具**。它让团队所有人知道「本季度最重要的事是什么」，以及「我们怎么判断自己做到了」。

OKR 在中国互联网企业落地失败最常见的两个原因：
1. **写成了 KPI**：把任务（做什么）当成关键结果（验证什么）
2. **形式主义**：制定完就放进文档，季度结束对着结果打分了事

这两个问题都可以通过正确的 OKR 方法论避免。

## 核心概念（Key Concepts）

### OKR 的基本结构

```
Objective（目标）：我们想去哪里？
  └─ Key Result 1（关键结果 1）：怎么衡量我们到了？
  └─ Key Result 2（关键结果 2）：怎么衡量我们到了？
  └─ Key Result 3（关键结果 3）：怎么衡量我们到了？
```

**目标（O）的特征：**
- 激励性：读到它时让人想起床工作
- 定性：描述方向，不带数字
- 有时限：通常以季度为单位
- 可达性边界：70% 完成率算正常（不是 KPI）

**关键结果（KR）的特征：**
- 可衡量：有具体数字
- 验证目标：能证明 O 已经实现
- 结果导向：不是任务或行动项
- 数量：每个 O 对应 2-4 个 KR

### OKR vs KPI：最关键的区别

| 维度 | OKR | KPI |
|------|-----|-----|
| 目的 | 设定方向、对齐聚焦 | 监控绩效、评估结果 |
| 制定者 | 自下而上 + 自上而下 | 通常自上而下 |
| 挑战性 | 设计上有延伸性（70% = 好） | 通常 100% 完成为目标 |
| 是否考核 | 不直接挂钩薪酬 | 通常挂钩薪酬 |
| 公开性 | 全公司可见 | 不一定 |

### OKR 与中国企业 KPI 文化的对接

中国企业通常有根深蒂固的 KPI 文化。引入 OKR 时面临的挑战：

- **「OKR 是不是新瓶装旧酒的 KPI？」** — 最常见质疑，需要管理层明确 OKR 不直接挂绩效考核
- **「完成 70% 不会被批吗？」** — 需要建立心理安全感，鼓励设定有野心的目标
- **「我们已经有月度 KPI 了，OKR 放哪？」** — 建议 OKR 管战略方向（季度），KPI 管运营基线（月度），两者并行但不重叠

### OKR 层级结构

```
公司级 OKR（CEO/创始人制定）
   └─ 事业部级 OKR（VP/总监对齐公司 OKR）
         └─ 团队级 OKR（产品/技术/运营等）
               └─ 个人 OKR（可选，大公司较少用）
```

对齐原则：下层 OKR 的实现应该**贡献**（而非简单复制）上层 OKR 的关键结果。

### 反模式（Anti-Patterns）

❌ **把任务写成 KR**
- 错误：「完成用户中心改版上线」
- 正确：「用户中心改版后，核心操作完成率从 60% 提升到 80%」

❌ **目标缺乏激励性**
- 错误：「维持产品稳定运营」
- 正确：「让用户在关键时刻第一个想到我们」

❌ **KR 数量过多**
- 错误：一个 O 对应 8 个 KR（什么都想要等于什么都没聚焦）
- 正确：每个 O 控制在 2-4 个 KR，优先级清晰

❌ **季度结束才看 OKR**
- 错误：制定完放进文档，复盘时才拿出来对分
- 正确：双周 Check-in，实时追踪进展，及时调整

## 应用方法（Application）

### 第一阶段：制定 OKR（第 1-2 周）

**步骤 1：确认本季度最重要的问题**

在写 O 之前，先回答这个问题：
> 「如果这个季度只能做好一件事，那件事是什么？为什么？」

工具：参考上层 OKR、上季度复盘结论、公司战略重点

**步骤 2：起草目标（O）**

用以下句型起草：
```
「本季度，我们要[方向性描述]，让[谁]感受到[什么价值/变化]。」
```

检验标准：
- [ ] 读起来让人兴奋（而非「维持 xxx」或「继续做 xxx」）
- [ ] 不含数字（数字放在 KR 里）
- [ ] 挑战性合理：完全实现概率约 50-70%

**步骤 3：制定关键结果（KR）**

针对每个 O，问：「我们怎么知道这个 O 真的实现了？」答案就是 KR。

KR 写作公式：
```
将 [指标名称] 从 [当前值] 提升/降低到 [目标值]
```

示例：
- 「将新用户 7 日留存率从 23% 提升到 35%」
- 「将 P0 Bug 平均修复时长从 48 小时降低到 12 小时」
- 「将月活跃用户中付费转化率从 3.2% 提升到 5%」

检验标准：
- [ ] 有明确的起始值和目标值
- [ ] 有单一、可量化的指标
- [ ] 结果导向（不是「完成 xxx 功能」这样的任务）
- [ ] 实现后能证明 O 达成

**步骤 4：评估 OKR 整体质量**

对每组 O + KRs 回答：
1. 如果所有 KR 都实现了，O 一定实现了吗？（若否，KR 遗漏了关键维度）
2. 如果 O 实现了，是否一定意味着所有 KR 都完成了？（若否，KR 可能过于保守）
3. 这组 OKR 和上层 OKR 有清晰的贡献关系吗？

---

### 第二阶段：对齐（第 2-3 周）

**横向对齐：消除跨团队依赖冲突**

产品团队的 OKR 通常需要技术、运营、市场配合。对齐会议议程：

1. 各团队展示 OKR 草稿（5 分钟/团队）
2. 识别依赖关系：「你的哪个 KR 需要我们的支持？」
3. 明确接口：谁做什么，什么时候完成
4. 标记风险：存在资源冲突的 KR 需要 leader 决策

**纵向对齐：确保支撑关系**

与上级 1:1 对齐：
- 「我的 OKR 如何支撑你的 OKR？」
- 「你认为我遗漏了哪些重要方向？」
- 「哪个 O 的优先级最高？」

---

### 第三阶段：追踪（每双周）

**双周 Check-in 格式（15 分钟）**

对每个 KR 更新：

```markdown
KR：将新用户 7 日留存率从 23% 提升到 35%

当前值：28%（目标 35%，进度 50%）
置信度：🟡 中（原计划的新手引导改版延期了）
本周进展：完成了引导流程原型设计
下周计划：完成 A/B 测试方案，提测开发
风险：开发资源被拉去做其他需求，需升级
```

置信度信号灯：
- 🟢 高（预计能达到或超越目标）
- 🟡 中（有风险，但可以追）
- 🔴 低（当前轨迹无法完成，需要干预）

**何时需要调整 OKR**

OKR 一旦制定不应轻易修改，但以下情况例外：
- 外部环境剧变（政策、市场突变）
- 依赖的上游资源被砍（团队裁员、预算削减）
- 发现原有假设根本性错误

调整时需记录原因，避免把「没完成」包装成「策略调整」。

---

### 第四阶段：季度复盘

**复盘会议结构（60-90 分钟）**

**1. 评分（15 分钟）**

对每个 KR 打分（0-1.0 分）：
- 0.7-1.0：超出预期
- 0.4-0.6：有进展但未完成
- 0.0-0.3：几乎没有推进

注意：0.7 分就是好成绩——OKR 设计上有延伸性。不要用完成率来评价 OKR 的价值。

**2. 分析根因（30 分钟）**

对每个 KR（尤其是低分的）回答：
- 是目标定得太激进？还是执行有问题？
- 哪些假设被证伪了？
- 下个季度我们应该继续、停止或开始什么？

**3. 经验输出（15 分钟）**

```markdown
本季度做得好的 3 件事：
1.
2.
3.

下季度要改变的 3 件事：
1.
2.
3.

未解决的核心问题（带入下季度 OKR 讨论）：
-
```

## 示例（Examples）

### 示例：某内容 App 产品团队季度 OKR

**公司级 O（上层）：** 成为用户每日必用的信息获取入口

---

**产品团队 OKR：**

**O1：让新用户在首周就感受到产品的独特价值**

KR1：新用户 7 日留存率从 23% 提升到 35%
KR2：新用户首次看到「感兴趣内容」的时间从 平均 4 分钟缩短到 90 秒
KR3：新用户 3 日内完成关注至少 3 个话题的比例从 15% 提升到 40%

**O2：让核心用户有理由每天回来**

KR1：日活/月活比（DAU/MAU）从 0.28 提升到 0.38
KR2：用户平均每日启动次数从 1.8 次提升到 2.5 次
KR3：Push 通知点开率从 4.2% 提升到 7%

---

**质量评估：**
- ✅ O1 有激励性（「感受到独特价值」）
- ✅ KR 全部有基准值和目标值
- ✅ KR 描述结果，不是任务
- ✅ KR 实现后可证明 O 达成
- ✅ 与公司级 O「每日必用」形成贡献关系

---

### 反例：被说成 KPI 的 OKR（错误示范）

**O：完成产品迭代计划**（❌ 没有方向性，不激励人）

KR1：完成用户中心改版上线（❌ 这是任务，不是结果）
KR2：完成 API 性能优化（❌ 技术任务，无用户视角）
KR3：推进 3 个新功能开发（❌「推进」不可量化）

**修改后：**

**O：让每个用户都能轻松管理自己的账号和数据**

KR1：用户中心核心操作（修改密码、绑定手机等）成功完成率从 72% 提升到 92%
KR2：用户对账号安全功能的满意度 NPS 从 +12 提升到 +35
KR3：API 超时导致的用户报错率从 0.8% 降低到 0.1%

## 常见陷阱（Common Pitfalls）

### 陷阱 1：「OKR 成为第二套 KPI」

**症状：** OKR 制定完就直接挂绩效，员工为了「安全」把目标写得保守

**后果：** 失去 OKR 的延伸性精神，退化为打分工具

**解法：** 管理层明确声明「OKR 不直接决定绩效评级」；单独建立 OKR 文化，让 0.7 分成为值得骄傲的结果

---

### 陷阱 2：「关键结果写成行动清单」

**症状：** KR 全是「完成 xxx 功能」「上线 xxx 模块」

**后果：** 即使全部完成，也不知道是否真正解决了用户问题

**解法：** 区分 Output（产出）和 Outcome（结果）。OKR 追踪 Outcome，任务看板追踪 Output

---

### 陷阱 3：「OKR 孤岛」

**症状：** 各团队 OKR 写完没有横向对齐，实际执行时发现互相依赖但没有协商

**后果：** 季度结束 A 团队完成了但 B 团队的 KR 因为等 A 而落空

**解法：** 制定完 OKR 后强制跨团队对齐会议，标记所有跨团队依赖，由双方团队 leader 签字确认

---

### 陷阱 4：「年度 OKR + 季度执行脱节」

**症状：** 年初制定了年度 OKR，但季度 OKR 和年度 OKR 关系不大

**后果：** 季度结束了，年度目标没有推进

**解法：** 季度 OKR 制定时，从年度 OKR 反推「这个季度要推进到哪个里程碑才算向年度目标前进了」

# 用户故事示例

## 示例1：好的用户故事

```markdown
### 用户故事 042：

- **摘要：** 为试用用户提供微信一键登录，降低注册摩擦

#### 用例（用户故事三段式）：
- **作为** 首次访问应用的试用用户
- **我想要** 用微信扫码一键登录
- **以便** 我不用输入手机号和验证码就能快速体验产品

#### 验收标准（验收标准三步法）：

- **场景：** 首次试用用户通过微信OAuth登录
- **假如：** 我在登录页面
- **并且假如：** 我手机上已登录微信
- **并且假如：** 页面上可见"微信登录"按钮
- **当：** 我点击"微信登录"按钮并扫码授权
- **那么：** 我自动登录并跳转到新手引导页面
```

**为什么这条故事好：**
- 角色具体（"首次访问应用的试用用户"，不是笼统的"用户"）
- 动作清晰（"微信扫码一键登录"，不是"更好的登录"）
- 动机明确（"不用输入手机号和验证码"，揭示了真实痛点）
- 验收标准可测试（测试同学看完就知道怎么验证）
- 只有一个"当"和一个"那么"（范围适当）

## 示例2：差的故事（太模糊）

```markdown
### 用户故事 999：

- **摘要：** 改进登录体验

#### 用例（用户故事三段式）：
- **作为** 用户
- **我想要** 更好的登录
- **以便** 我可以使用应用

#### 验收标准（验收标准三步法）：

- **场景：** 用户登录
- **假如：** 我想登录
- **并且假如：** 我有账号
- **当：** 我登录
- **那么：** 体验变好了
```

**为什么这条故事失败：**
- "用户"太泛了——试用用户？回头客？管理员？
- "更好的登录"不是具体动作——什么叫"更好"？
- "使用应用"不是有意义的动机——所有用户都想用应用
- 验收标准完全不可测试——"体验变好了"无法量化

**怎么改：**
- 收窄角色："首次注册的试用用户""没装密码管理器的回头客"
- 定义动作："用微信扫码登录""通过邮箱重置密码"
- 明确结果："不用记密码""30秒内完成登录"
- 验收标准可证伪："那么我在2秒内跳转到首页"

## 示例3：需要拆分的故事

```markdown
### 用户故事 100：

- **摘要：** 管理购物车

#### 用例（用户故事三段式）：
- **作为** 购物者
- **我想要** 加商品、删商品、改数量、用优惠券、去结算
- **以便** 我能完成购买

#### 验收标准（验收标准三步法）：

- **场景：** 购物车管理
- **假如：** 我的购物车里有商品
- **当：** 我添加一件商品
- **那么：** 它出现在购物车里
- **当：** 我删除一件商品
- **那么：** 它从购物车消失
- **当：** 我修改数量
- **那么：** 数量发生变化
- **当：** 我使用优惠券
- **那么：** 折扣被应用
- **当：** 我去结算
- **那么：** 进入支付页面
```

**为什么需要拆分：**
- 多个"当"语句 = 多条独立故事
- 一次冲刺做不完这么大的范围
- 加商品 ≠ 用优惠券，结果互不相关

**怎么拆分：**
拆成5条独立故事——
1. "往购物车里加商品"
2. "从购物车删除商品"
3. "修改商品数量"
4. "使用优惠券"
5. "去结算进入支付"

每条故事都有聚焦的验收标准。

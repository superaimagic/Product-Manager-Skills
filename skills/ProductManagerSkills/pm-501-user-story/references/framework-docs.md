# 用户故事参考资料

## 核心框架

### INVEST原则
- **I**ndependent（独立的）：故事之间应该相互独立
- **N**egotiable（可协商的）：故事是对话的起点，不是合同
- **V**aluable（有价值的）：每个故事都应该为用户或业务提供价值
- **E**stimable（可估计的）：团队应该能够估算故事的大小
- **S**mall（小的）：故事应该在一个冲刺内完成
- **T**estable（可测试的）：故事必须有清晰的验收标准

### 用户故事三段式来源
源自 Mike Cohn 的《User Stories Applied》（2004），是全球敏捷团队的标准实践。

### 验收标准三步法来源
源自行为驱动开发（BDD）社区的 Gherkin 格式，已被 SAFe、LeSS 等规模化敏捷框架采纳。

## 常用术语

### Persona（用户角色）
对产品目标用户的半虚构描述，基于真实用户研究。

### Acceptance Criteria（验收标准）
定义一个功能被认为完成的具体条件，通常用"Given-When-Then"格式。

### Story Point（故事点数）
用于估算用户故事大小的相对度量单位。

## 中国团队实践建议

### 本土化适配
- 使用微信、支付宝等本土社交登录方式作为案例
- 结合钉钉、企业微信等协作工具场景
- 考虑中国用户的使用习惯和文化背景

### 跨团队沟通
- 确保产品、开发、测试使用统一的用户故事格式
- 定期进行故事梳理会议，保持团队对齐
- 使用飞书文档、腾讯文档等工具进行协作
